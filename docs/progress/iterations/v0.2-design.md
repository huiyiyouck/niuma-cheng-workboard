### IRC-001：数据库选型变更 SQLite → PostgreSQL
- **触发方**：Owner（服务器已有 PostgreSQL 16，不应另装 SQLite）
- **变更内容**：
  - 驱动：`better-sqlite3`（同步 API）→ `pg`（异步连接池）
  - 存储路径：`data/workboard.db`（本地文件）→ PostgreSQL `workboard` 库（服务器端）
  - 连接配置：通过 `.env` 文件（已加入 .gitignore）管理 `PG_HOST/PG_PORT/PG_USER/PG_PASSWORD/PG_DATABASE`
  - 部署：本地开发用 SSH 隧道（`localhost:15432` → 服务器 5432）；服务器部署直接连 `localhost:5432`
- **影响范围**：
  - `src/server/db.js`：完全重写，从 better-sqlite3 同步 API 改为 pg Pool 异步 API
  - `src/server/sync/claude-sync.js`：全部查询改为参数化（`$1, $2...`）+ 事务用 `BEGIN/COMMIT/ROLLBACK` + `ON CONFLICT DO UPDATE/NOTHING`

  - `package.json`：移除 `better-sqlite3`，新增 `pg`
- **设计文档影响**：§3 数据模型（表结构不变，仅类型调整 SMALLINT）和 §4 API 契约（响应格式从对象改为数组 row）仍成立，但实现细节需以代码为准
- **状态**：已落地

## 1. 概述

v0.2 在 v0.1「本地 Node 只读聚合服务」基础上升级为「带写入的本地工作台」，新增第 5 个一级视图「项目会话」。

核心架构变更：

- **从纯文件只读升级为文件读取 + SQLite 持久化**：引入 `better-sqlite3` 存储角色-会话映射关系和 Claude Code 会话缓存
- **从单一聚合 API 升级为多 API 服务**：新增 4 组 API（会话列表、对话内容、映射 CRUD、迭代时间轴）
- **新增会话同步引擎**：定时扫描 `~/.claude/projects/` 下 JSONL 文件，增量同步到 SQLite 缓存层
- **新增迭代时间轴解析器**：解析 INDEX.md 版本列表 + vX.Y.md 阶段门禁状态

核心原则（沿用 v0.1）：

- **只读真源不变**：`projects.config.json`、各项目 `docs/progress/`、coordination 真源仍然只读
- **配置驱动**：生态分层结构由 `projects.config.json` 的 `level` 字段驱动
- **项目级隔离**：单项目异常不阻断其他项目
- **稳定接口**：后端解析成稳定 JSON，前端只消费模型

### 1.1 技术选型决策

| 决策项 | 选型 | 理由 |
|--------|------|------|
| SQLite 驱动 | `better-sqlite3` | 同步 API，单用户本地场景简单直接；性能好；Node 原生模块生态成熟 |
| 数据库文件路径 | `data/workboard.db` | 不入库（`.gitignore`）；可配置（环境变量 `WORKBOARD_DB_PATH`） |
| Markdown 渲染 | `react-markdown` + `remark-gfm` + `rehype-highlight` | shadcn 生态兼容；支持 GFM；语法高亮；禁用原始 HTML 防 XSS |
| 前端状态层 | 独立 `useProjectSession` hook | 不污染 v0.1 ViewModel Context；项目会话视图状态独立管理 |
| 后端路由 | 保持手工路由（`node:http`） | v0.1 已有模式；API 数量可控（≤10 个）；不引入框架 |

### 2.4 迭代时间轴（只读解析）

#### 第一层：版本列表

从 INDEX.md「版本列表」表格解析。表格列名映射：

| 表格列 | 字段 | 说明 |
|--------|------|------|
| 版本 | version | 如 v0.1、v0.2 |
| 迭代记录 | recordPath | 相对路径，如 iterations/v0.1.md |
| PRD | prdPath | |
| UI | uiPath | |
| 设计文档 | designPath | |
| Summary | summaryPath | |
| 状态 | status | 解析为枚举 |

**状态解析规则**：
- 包含「✅」或「已完成」→ `"completed"`
- 包含「🔵」或「进行中」→ `"in_progress"`
- 包含「📋」或「规划中」→ `"planned"`
- 包含「待创建」→ `"planned"`（产物未创建）
- 其他 → `"unknown"`

#### 第二层：迭代详情

从 `vX.Y.md` 解析阶段门禁状态。

**解析策略**：
1. 定位「## 阶段门禁」二级标题
2. 遍历其下所有三级标题（`### XXX 阶段` / `### XXX`），从三级标题提取阶段名
3. 每个三级标题下找第一个表格，按**列名匹配**「阶段状态」列（不按固定列位置），提取状态值
4. 完成日期从表格中「Review 结果」列或阶段状态文本中提取（best-effort，解析不到为 null）
5. 产出文档链接从 INDEX.md 版本列表表格对应列提取（不在 vX.Y.md 中解析）

**阶段映射表**（标准迭代的 6 个阶段 + 已知变体）：

| 阶段名匹配 | 责任角色 | 说明 |
|------------|----------|------|
| PRD 阶段 / PRD | PM | 标准阶段 |
| UI 方案阶段 / UI | PM | UI 先行变体有 |
| 设计阶段 / 设计 | Architect | 标准阶段 |
| 实现阶段 / 实现 | Developer | 标准阶段 |
| 测试阶段 / 测试 | Tester | 标准阶段 |
| 部署就绪检查 / 部署就绪检查 / 部署 | DevOps | 标准阶段 |
| 迭代关闭 / 关闭 | PM | 标准阶段 |
| 收尾 / 收尾归档 | 各角色 | 标准阶段 |

**未知阶段兜底**：三级标题匹配不到上表时，阶段名用原文，责任角色为 `"unknown"`，状态照常解析，不跳过。

**阶段状态中文 → 枚举映射**：

| 中文状态 | 枚举值 |
|----------|--------|
| 已定稿 / 已完成 / ✅ | finalized |
| Review 中 / Review中 / 进行中 / 🔵 / 待Review | in_progress |
| 未开始 / 待启动 | not_started |
| 阻塞 / 待修正 | blocked |

**阶段状态枚举**：`"finalized"`（已定稿）/ `"in_progress"`（进行中）/ `"not_started"`（未开始）/ `"blocked"`（阻塞）

**产出内容摘要提取（三级降级）**：
1. **最低保证形态**（必须展示）：阶段名 + 责任角色 + 完成日期 + 产出文档链接（从 INDEX.md 版本列表表格提取）
2. **一句话摘要**（best-effort）：尝试从 `vX.Y-summary.md` 的首段或 `vX.Y.md` 的「概述」段落提取；解析不到就不显示
3. **详细描述**：v0.2 不做，走「产出文档链接」和「查看当时的对话」钻取

## 4. 核心流程

### 4.1 服务启动流程

1. 读取 `projects.config.json`，校验 config schema（含 v0.2 新增 `level` / `ecosystem` 校验）
2. **校验 `ecosystem.root_session_id` 不存在**（PRD §6.1 定稿）；存在则报错退出
3. 初始化 SQLite 数据库（`data/workboard.db`）：
   - 文件不存在 → 创建并执行建表 DDL
   - 文件存在 → 执行 `PRAGMA table_info` 检查表结构，缺失则补建
4. 启动会话同步引擎（首次全量同步 + 定时增量同步）
5. 启动 HTTP 服务

### 4.2 会话同步引擎

**同步模式**：

| 场景 | 策略 |
|------|------|
| 首次启动 | 遍历 `~/.claude/projects/` 所有 `.jsonl` 文件，全量解析入库 |
| 定时增量（每 30 秒） | 1. 扫描目录发现新文件 → 全量解析<br>2. 对比已有文件 `file_mtime`，有变更的触发增量读取 |
| 手动触发 | 用户调用 `POST /api/sync` 接口（对话视图「刷新」按钮） |

**增量同步算法**：

```
对每个 .jsonl 文件：
  1. 查 claude_sessions 表获取 last_byte_pos 和 file_mtime
  2. 获取当前文件 stat.mtime 和 stat.size
  3. 若 file_mtime 未变 → 跳过
  4. 若 file_mtime 变了但 size < last_byte_pos → 文件被截断，全量重读（last_byte_pos = 0）
  5. 若 file_mtime 变了且 size >= last_byte_pos → 从 last_byte_pos 开始追加读取
  6. 解析新增行，插入 claude_messages 表
  7. 更新 claude_sessions 的 last_byte_pos、file_mtime、last_sync_at、消息计数
```

**Claude Code 配置目录路径**：
- 默认 `~/.claude/`
- 可通过环境变量 `CLAUDE_HOME` 配置
- 生产环境需确认服务进程有读取权限

### 4.3 JSONL 解析与 content 提取规则

**JSONL 每行是一个 JSON 对象**，按 `type` 字段区分：

| type | 处理方式 |
|------|----------|
| `user` | 提取为 `claude_messages` 记录，`role = "user"` |
| `assistant` | 提取为 `claude_messages` 记录，`role = "assistant"` |
| `custom-title` | 更新 `claude_sessions.title` |
| `system` / `attachment` / `queue-operation` / `mode` / `last-prompt` | 跳过（不存入 messages 表） |

**content 提取规则**：

| 消息类型 | content 来源 | 提取策略 |
|----------|-------------|----------|
| user | `message.content` | 若为字符串，直接存储；若为数组，拼接所有 text block |
| assistant | `message.content` | 必为数组。**只拼接 `type === "text"` 的 block**；`thinking` 和 `tool_use` 不拼入 content |

**tool_name 和 tool_input 提取**：
- assistant 消息中 `type === "tool_use"` 的 block：提取 `name` → `tool_name`，提取 `input` → `tool_input`（JSON 序列化）
- 一条 assistant 消息可能有多个 tool_use block：只取第一个的 `name` 和 `input`（v0.2 简化处理）

**title 提取优先级**：
1. `custom-title` 事件的 `customTitle` 字段
2. 首条 user 消息 `content` 前 50 字符
3. fallback：`"未命名会话"`

### 4.4 角色自动识别规则

> **v0.2 实现决策**：角色自动识别作为辅助标签实现，但**不作为核心路径**。识别结果仅展示在会话列表中，最终映射以用户手动配置为准。

识别策略：
1. **精确匹配**（置信度 1.0）：首条 user 消息匹配 `你是\s*(PM|Architect|Developer|DevOps|General|产品经理|架构师|开发工程师|运维部署工程师|通用助手)`
2. **关键词加权**（置信度 0.5~0.9）：扫描前 5 条 user 消息，按关键词命中数加权评分
3. **无法识别**：`detected_role = "Unknown"`，`role_confidence = 0`

### 4.5 映射配置流程

```
用户点击「配置映射」
  → 弹出三步弹窗（或参谋长单步弹窗）
  → 步骤 1：选择项目（3 个子项目）
  → 步骤 2：选择角色（5 个角色）
  → 步骤 3：选择会话（GET /api/sessions 获取列表）
  → 确认 → PUT /api/mappings
  → 后端 upsert role_mappings 表
  → 返回 mapping 对象
  → 前端刷新角色卡片状态
```

### 4.6 迭代时间轴解析流程

```
GET /api/projects/:projectId/timeline
  1. 从 config 查找项目，解析路径
  2. 读取 {projectPath}/docs/progress/INDEX.md
  3. 解析「版本列表」表格 → TimelineVersion[]
  4. 解析「当前项目状态」→ 提取 currentVersion
  5. 对每个版本：
     a. 读取 {projectPath}/docs/progress/iterations/vX.Y.md
     b. 定位「## 阶段门禁」二级标题
     c. 遍历其下三级标题（「### XXX 阶段」）提取阶段名
     d. 每个三级标题下第一个表格按列名匹配「阶段状态」列 → TimelineStage[]
     e. 对已完成的版本，尝试提取 summary（三级降级）
  6. 组装响应
```

**异常处理**：
- INDEX.md 不存在 → `404` + 错误提示
- vX.Y.md 不存在 → 该版本的 `stages` 为空数组，不阻断其他版本
- 解析异常 → 单版本降级，不影响其他版本

### 4.7 生产数据源同步

**不在 v0.2 后端代码中实现**，由 DevOps 在部署阶段落地 cron/systemd timer 脚本：

  cd "$repo"

## 7. 风险和取舍

| 风险 | 严重度 | 处理 |
|------|--------|------|
| `better-sqlite3` 原生编译环境依赖 | 中 | Node 原生模块，需确保目标环境有编译工具链；`npm install` 时自动编译 |
| JSONL 文件结构变更（Claude Code 升级） | 中 | 解析器版本兼容；未知 type 跳过不报错；数据库可随时从原始文件重建 |
| 首次全量同步性能（会话数量多时） | 中 | 30 秒定时增量同步；首次同步进度提示；数据库可随时重建 |
| SQLite 缓存与 JSONL 短暂不一致（≤30 秒） | 低 | 对话视图显示「最后同步时间」+ 手动刷新按钮 |
| 迭代时间轴解析复杂度 | 中 | 三级降级策略；单版本解析失败不阻断其他版本 |
| 角色自动识别准确率 | 低 | 仅辅助标签，最终以用户手动映射为准 |
| 生产环境 `~/.claude/` 路径与权限 | 中 | 环境变量 `CLAUDE_HOME` 可配置；部署时确认权限 |

取舍：

- **不引入 ORM**：SQL 语句直接写，表少（3 张）场景 ORM 是过度设计
- **不做服务端分页**：单用户本地场景，会话数 ≤几百，前端本地分页足够
- **角色自动识别简化**：只做首条消息精确匹配 + 关键词加权，不引入 NLP
- **生产数据源同步不在后端代码**：由 DevOps cron 脚本落地，后端只读取同步状态
