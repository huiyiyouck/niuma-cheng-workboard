## 1. 迭代目标

交付「项目管理工作台」第一版：一个本地可运行的**只读**看板，配置驱动地聚合牛马程生态各项目（workflow / xiaobao / ai / coordination / workboard）的工作流状态、接入诊断、跨项目协作（需求 / 谁等谁 / 沟通 / BCR）和跨项目待办，给 Owner 一处掌握全局的入口。

第一版严格只读、不回写任何被监控项目；它是「项目管理统一中枢」愿景的第一步（看清），后续再长出管理能力（管得了）。

## 2. 用户故事

- **US-1 项目总览**：作为 Owner，我想在工作台一屏看到所有项目的当前迭代 / 模式 / 阶段 / 阻塞 / 下一步，以便不用逐个翻 `INDEX.md` 就掌握全局。
- **US-2 项目详情钻取**：作为 Owner，我想点项目卡从右侧抽屉看它的角色状态、待办和入口，以便深入单个项目又不跳页、看完即回。
- **US-3 接入诊断**：作为 Owner，我想看到每个配置项目的路径解析、接入状态、文件检查和错误，以便排查哪个项目没接入或读取异常。
- **US-4 跨项目协作**：作为 Owner，我想看到跨项目需求池、谁等谁、沟通和 BCR，以便掌握项目间依赖与协作进展。
- **US-5 跨项目待办汇总**：作为 Owner，我想在工作台集中看到各项目未完成的跨任务待办，以便统筹优先级。
- **US-6 部署入口**：作为 Owner，我想看到各项目的线上地址并一键打开，以便快速访问已部署的项目。
- **US-7 配置驱动接入**：作为 Owner，我想通过改 `projects.config.json` 增减监控项目，以便不改代码就纳入或移除项目。
- **US-8 自动刷新**：作为 Owner，我想看板按配置间隔（默认 60s）自动拉取最新状态，以便不手动刷新也看到最新。

## 3. 验收标准

> 格式「当…时，应该…」，每条可验证，覆盖正常 + 异常。

- **AC-1.1**（US-1）当打开工作台，应默认展示 config 中所有启用项目的卡片，并在顶部展示统计条、跨项目协作横条、跨项目待办汇总。
- **AC-1.2**（US-1，异常）当某项目读取失败时，该卡片应显示「读取异常」徽标与错误摘要，且不影响其他卡片正常渲染。
- **AC-2.1**（US-2）当点击项目卡时，应从右侧滑出该项目详情抽屉（迭代信息、角色、待办、入口、配置路径）；关闭后回到原视图与滚动位置。
- **AC-2.2**（US-2，边界）当 `business`/`workboard` 项目缺少 `INDEX.md` 时，应显示「未接入团队工作流」；`coordination`/`workflow-source` 不因缺 `INDEX.md` 被判为未接入或异常。
- **AC-3.1**（US-3）当进入接入诊断时，应列出所有配置项目的 id / name / kind / 解析路径 / 接入状态 / 文件检查 / 解析器 / 最近读取时间 / 错误摘要。
- **AC-3.2**（US-3，异常）当项目配置非法（路径不存在 / id 重复 / kind 不支持 / 环境变量缺失）应标为「配置异常」，当路径存在但文件不可读应标为「读取异常」，两者分别标识。
- **AC-4.1**（US-4）当进入跨项目视图时，应展示 4 个 Tab（需求池 / 谁等谁 / 沟通 / BCR）；需求池与 BCR 默认只显进行中，可切换「已关闭 / 全部」。
- **AC-4.2**（US-4，异常）当 coordination 真源不可读时，跨项目区应降级提示而非整页崩溃；无活跃阻塞时「谁等谁」显示空态。
- **AC-5.1**（US-5）当打开工作台，待办汇总应聚合各项目 `INDEX.md`「跨任务待办」中**未完成**的条目，展示优先级 / 待办 / 归属项目 / 归属角色 / 状态，且为只读（不可勾选、编辑）。
- **AC-6.1**（US-6）当项目配置了线上 `url` 时，部署页应展示可点击入口并在新标签打开；未配置 `url` 的项目应灰显「未配置线上地址」。
- **AC-7.1**（US-7）当在 `projects.config.json` 增减项目并满足该 kind 接入规则后，看板应相应增减监控项目，无需改动代码。
- **AC-7.2**（US-7，异常）当某项目配置字段缺失或非法时，应仅该项目标记异常并提示，不影响其他项目与整体渲染。
- **AC-8.1**（US-8）当看板运行时，应按 `refresh.interval_seconds`（默认 60）自动拉取最新数据刷新视图。
- **AC-8.2**（US-8，异常）当自动刷新中单个项目读取失败时，错误应隔离在该项目级，不阻断其他项目与整页刷新。

## 4. 范围边界

**本迭代做：**
- 上述 US-1~US-8 全部；四视图（工作台 / 部署 / 接入诊断 / 跨项目）+ 项目详情右抽屉 + 跨项目协作横条 + 待办汇总；配置驱动接入与接入校验；60s 自动轮询；前端复用 `frontend/` 原型基线 + 新建本地 Node 后端做只读读取与聚合。

**本迭代不做：**
- 不在页面编辑 Markdown、不回写任何被监控项目文件；
- 待办、需求、状态均**只读**，不可勾选 / 编辑 / 新建；
- 不做登录 / 权限、不做数据库；
- 不在页面保存配置变更（config 仍由 Owner 手改文件）；
- 深色模式仅保留 token / 视觉位，不做实际切换逻辑（可后置）；
- 不做复杂排期 / 甘特 / Jira 化；
- 不强制实现读取快照（单项目失败降级即可）。

## 5. 前置依赖
- 各项目 `docs/progress/INDEX.md`、`roles/` 可读；coordination 真源（`REQUESTS.md` / `STATUS.md` / `contracts/` / `communications/`）可读。
- `projects.config.json` 已存在（含 5 项目 + refresh）；部署页需为项目新增 `url` 字段。
- 前端基线已落 `frontend/`（React 18 + Vite + Tailwind v4 + shadcn）；需新建本地 Node 后端读文件 + 提供聚合 API。
- UI 方案 `v0.1-ui.md`（已定稿）、数据规格 `v0.1-ui-data-spec.md`。

## 6. 风险和开放问题
- **跨项目需求归属识别**：`REQUESTS.md` 承接方为自由文本（如 `ai · PM(ck)`），需按 token / 词边界识别项目 id，避免 `ai` 这类短 id 被误命中（立项 §7.3）——实现阶段落实。
- **角色状态解析**：各项目 `roles/` 文件形态不一（`{role}-current/summary/archive`、单 `{role}.md`），解析需兼容并降级（暂无日志 / 未启用 / 读取异常）。
- **无 INDEX 的 kind**：`workflow-source` / `coordination` 无迭代字段，卡片/横条按摘要降级展示，不可套业务卡模板。
- **后端读取性能**：每 60s 读多项目多文件，数据量小可整批读；若后续项目变多再评估缓存。
- **config `url` 字段**：当前配置未含 `url`，部署页依赖它，需补字段定义。

### R1 — Architect Review

- 结论：通过，带中风险后置到设计 / 实现阶段处理。
- Review 范围：各项目 `INDEX.md` / `roles/` / coordination 真源是否支撑 PRD 全部功能；接入校验、路径解析、状态枚举和跨项目数据聚合的可行性。
- 依据：
  - `projects.config.json` 已提供 5 个配置项目、`kind`、相对路径和 60s refresh 配置，可支撑配置驱动接入、路径解析、按 `kind` 分流和项目级错误隔离。
  - 普通项目的 `docs/progress/INDEX.md` 已提供当前迭代 / 模式 / 阶段 / 阻塞 / 下一步 / 跨任务待办等字段；`docs/progress/roles/` 存在时可支撑角色入口和最近角色状态，缺失时可降级为未启用 / 暂无日志。
  - coordination 真源中 `REQUESTS.md` 已同时包含普通跨项目需求池和 BCR 池，`STATUS.md` 包含谁等谁 / 阻塞 / 下一步，`contracts/` 与 `communications/` 文件存在，可支撑跨项目 4 Tab 和协作横条。
  - UI 数据规格已经明确 `workflow-source` / `coordination` 不套普通迭代卡模型，降低了无 `INDEX.md` 项目被误判异常的架构风险。
- 问题：
  - 严重度：中。PRD 把 Markdown 表格作为多个核心数据源（跨任务待办、REQUESTS、BCR、STATUS 摘要），但未定义解析失败、列名漂移、表格为空、完成态文案不统一时的接口契约。设计阶段需要把“可解析字段 / 原文兜底 / 错误摘要”明确到数据模型。
  - 严重度：中。跨项目需求归属识别已列为风险，`ai` 这类短 id 必须用配置项目 id、项目名、词边界和显式字段共同匹配，不能做简单子串包含。
  - 严重度：中。跨任务待办的“未完成”判断依赖各项目 `INDEX.md` 的自由文本状态，需定义完成态白名单或终态枚举；否则容易把“已完成（含说明）”误纳入待办汇总。
  - 严重度：低。部署页依赖 `projects.config.json` 的可选 `url` 字段，当前配置未包含；实现前需要补充字段 schema，并确认无 `url` 只灰显、不触发读取异常。
- 建议：
  - 设计阶段定义稳定后端返回模型：`ProjectSummary`、`ProjectDiagnostics`、`RoleSummary`、`CrossProjectRequest`、`BcrItem`、`TodoItem`、`SnapshotResult`，并把原始 Markdown 解析错误收敛为项目级 / 区域级 `read_error`。
  - Markdown 解析优先按标题锚点 + 表格列名解析；解析失败时保留原文片段和错误摘要，前端展示降级状态，不阻断其他区域。
  - 接入状态沿用 `integrated` / `not-bootstrapped` / `config-error` / `read-error` / `disabled` 作为接口枚举，UI 再映射中文徽标。
  - `/api/snapshot` 类整批接口应返回 `generatedAt`、`refreshIntervalSeconds` 和项目级结果数组；60s 轮询中单项目失败不得影响整批其他项目。

### R1 — Developer Review

- 结论：通过，带中风险后置到设计 / 实现阶段处理。
- Review 范围：`frontend/` 原型基线复用度、后端只读读取 / 聚合 / 60s 轮询的工程成本与可行性、需求是否可实现与验收是否可验证。
- 依据：
  - `frontend/` 已存在 React + Vite + Tailwind v4 + shadcn/Radix 组件基线，PRD 选择复用原型而不是重新做静态 HTML，工程路线已经收敛。
  - US-1~US-8 均有对应 AC，覆盖正常、异常和边界场景；项目级读取失败、coordination 降级、无 `INDEX.md` kind 降级、无 `url` 灰显等关键异常都可实现并可测试。
  - 后端只读读取 Markdown / config 并聚合为前端 API，第一版无数据库、无登录、无写回，复杂度可控。
  - 60s 轮询可以通过整批快照接口或等价聚合接口实现，单项目错误隔离符合前端渲染模型。
- 问题：
  - 严重度：中。PRD 已要求“本地可运行”，但尚未明确根目录启动方式、前后端并行方式和端口约定；当前只有 `frontend/package.json`，后端与根 `npm start` 需在设计 / 实现阶段定清。
  - 严重度：中。验收标准要求配置驱动增减项目，但未明确 `projects.config.json` schema、非法字段校验错误格式和 `url` 字段类型；实现阶段需要先定义配置解析契约。
  - 严重度：中。跨任务待办、REQUESTS、BCR、STATUS 都依赖 Markdown 表格 / 标题解析；PRD 已列风险，但实现需要明确“解析失败时返回原文兜底还是仅错误摘要”，否则前端不同区域的降级状态会不一致。
  - 严重度：低。`frontend/package.json` 中 React / ReactDOM 目前在 peerDependencies 且标 optional；若作为本项目真实前端交付，建议实现阶段把运行依赖整理为可稳定 `install + build` 的形式。
- 建议：
  - 设计阶段先定义最小后端 API：`GET /api/snapshot`、`GET /api/config/diagnostics` 或等价接口，并明确响应模型、错误枚举、刷新时间字段。
  - 在实现第一步补齐根目录工程脚本，明确一条 Owner 可用的本地启动命令，例如根目录 `npm start` 同时启动后端并服务前端构建产物，或文档化 dev 模式。
  - 配置解析使用显式 schema 校验，至少覆盖 `id` 唯一、`name`、`path`、`kind`、`enabled`、可选 `url`、`refresh.interval_seconds`。
  - Markdown 解析模块按“标题定位 + 表格列名”实现，解析失败返回区域级或项目级错误对象，前端只负责展示降级状态。
