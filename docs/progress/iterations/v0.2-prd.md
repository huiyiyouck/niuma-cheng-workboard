## 技术预研记录

### Spike-001 — Claude Code 会话数据读取可行性验证

- **预研时间**：2026-07-05
- **预研人**：Owner 直连验证
- **结论**：✅ **完全可行**。Claude Code 会话数据以 JSONL 格式存储在本地文件系统，可直接读取解析，无需依赖外部 API 或数据库。
- **数据位置**：`~/.claude/projects/-<编码后项目路径>/<sessionId>.jsonl`
- **项目路径编码规则**：路径中 `/` 替换为 `-`，整体前缀加 `-`。例如 `/Users/ck/Project/niuma-cheng` → `-Users-ck-Project-niuma-cheng`
- **验证证据**：
  - 本机共发现 **9 个项目目录**、**34 个会话**，其中 **32 个有完整对话**（用户消息 + 助手消息）
  - 总用户消息 **3,976 条**，总助手消息 **8,080 条**
  - 消息结构清晰，类型明确，可稳定解析
- **数据结构要点**：
  - **会话列表**：遍历 `~/.claude/projects/` 下所有 `.jsonl` 文件即可枚举所有会话
  - **会话元数据**：每条记录包含 `sessionId`、`cwd`（项目路径）、`gitBranch`、`timestamp`、`version`、`entrypoint`
  - **用户消息**：`type: "user"`，`message.content` 为字符串或 content block 数组
  - **助手消息**：`type: "assistant"`，`message.content` 为 block 数组，含 `text`（文本）、`thinking`（思考）、`tool_use`（工具调用）
  - **消息顺序**：通过 `parentUuid` 字段可构建对话链；按 `timestamp` 排序也可得到时间序列
  - **工具调用**：`type: "tool_use"` / `tool_result` / `direct`，可辅助理解对话上下文
- **对 v0.2 的影响**：
  - 原「高风险阻塞项」解除，对话视图（US-7/8/14）和配置弹窗会话下拉（US-3/4）的数据来源明确
  - 会话列表 API 实现路径：服务器端读取 `~/.claude/projects/` 目录，解析每个 JSONL 文件首条记录提取元数据
  - 对话内容 API 实现路径：按 `sessionId` 定位对应 JSONL 文件，筛选 `user` / `assistant` 类型记录，提取文本内容
  - 无需引入额外第三方依赖，纯文件系统读取即可
- **注意事项**：
  - 生产环境需考虑 Claude Code 配置目录的路径配置（本地 `~/.claude/`，生产可能不同）
  - 会话列表的「标题」/「摘要」需从首条用户消息中提取（前 N 字符作为会话标题）
  - `~/.claude/` 为用户目录，需确保服务进程有读取权限
  - JSONL 文件可能较大（几 MB），对话内容 API 建议支持分页或按需加载

## 1. 迭代目标

交付「项目管理工作台」第二版：从「只读聚合看板」升级为 **Web 端项目迭代 IDE**——在 v0.1 四视图（只读）基础上，新增第 5 个一级视图「项目会话」，实现**生态分层结构展示**、**角色会话映射配置**、**对话内容只读查看**和**迭代时间轴**，让 Owner 通过统一入口切换项目、映射角色会话、查看对话内容和迭代进度。

v0.2 严格只做「看 + 配置映射」，不做对话交互输入（v0.3）、不做增删改查（v0.3）、不做对话输入（v0.4）。

- **AC-E.5**v0.1 四视图（工作台/部署/接入诊断/跨项目）功能不受影响。

## 4. 范围边界

**本迭代做：**
- 生态分层结构展示（生态根目录视图 + 子项目视图 + 面包屑导航）
- 5 个角色的会话映射配置（三步弹窗）+ 参谋长席位映射（单步弹窗）
- 映射关系持久化到 SQLite 数据库
- 已映射角色/席位的对话内容只读查看（对话视图）
- 双层迭代时间轴（项目迭代总览轴 + 迭代详情轴，区分进行中/已完成两种展示模式）
- 阶段详情抽屉（产出文档 + 产出摘要 + 查看当时的对话）
- 左菜单顺序调整为工作台/项目会话/跨项目/部署/接入诊断
- 生产数据源自动同步（定时 git pull + 失败降级 + 同步状态展示）

**本迭代不做：**
- 不在页面发消息/输入对话（v0.3+ 只读，v0.4 做对话输入）
- 不做映射关系的增删改查完整 CRUD（v0.2 只做映射配置 + 查看映射，v0.3 做完整 CRUD）
- 不做会话内容的自动刷新（对话视图静态加载）
- 不做迭代时间轴的自动刷新（切换视图时重新读取）
- 不做工作流文档的在线编辑
- 不做深色模式实际切换
- 不做登录/权限
- 不做子项目卡片的排序/筛选
- 不做 git 冲突自动解决（冲突时标记异常，需人工处理）

### 6.2 角色会话映射（SQLite）

CREATE TABLE role_mappings (

### 6.3 Claude Code 会话缓存（SQLite + 增量同步）

**设计原则**：原始真源是 `~/.claude/projects/` 下的 JSONL 文件，数据库是缓存层，随时可从原始文件重建。

**核心表设计**：

```sql
-- 会话元数据表
CREATE TABLE claude_sessions (
  session_id    TEXT PRIMARY KEY,       -- Claude Code 会话 ID (UUID)
  file_path     TEXT NOT NULL,          -- JSONL 文件绝对路径
  project_path  TEXT,                   -- 所属项目路径（cwd 字段）
  git_branch    TEXT,                   -- Git 分支
  first_msg_at  TEXT,                   -- 首条消息时间
  last_msg_at   TEXT,                   -- 最后消息时间
  user_count    INTEGER DEFAULT 0,      -- 用户消息数
  assistant_count INTEGER DEFAULT 0,    -- 助手消息数
  tool_count    INTEGER DEFAULT 0,      -- 工具调用数
  title         TEXT,                   -- 会话标题（首条用户消息前 N 字）
  detected_role TEXT,                   -- 自动检测的角色（PM/Architect/Developer/DevOps/General/Unknown）
  role_confidence REAL,                 -- 角色检测置信度 0~1
  last_sync_at  TEXT NOT NULL,          -- 最后同步时间
  last_byte_pos INTEGER DEFAULT 0,      -- 上次读取到的文件字节位置（用于增量同步）
  file_mtime    REAL                    -- 文件最后修改时间（用于检测变更）
);

-- 对话消息表
CREATE TABLE claude_messages (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id  TEXT NOT NULL,            -- 所属会话 ID
  msg_uuid    TEXT NOT NULL,            -- 消息 UUID
  parent_uuid TEXT,                     -- 父消息 UUID（构建对话链）
  role        TEXT NOT NULL,            -- user / assistant
  content     TEXT NOT NULL,            -- 消息文本内容
  timestamp   TEXT NOT NULL,            -- 消息时间
  tool_name   TEXT,                     -- 工具调用名称（assistant 消息）
  tool_input  TEXT,                     -- 工具调用参数 JSON
  model       TEXT,                     -- 使用的模型（assistant 消息）
  stop_reason TEXT,                     -- 停止原因
  line_no     INTEGER NOT NULL,         -- 在 JSONL 文件中的行号
  UNIQUE (session_id, msg_uuid)
);

CREATE INDEX idx_messages_session_time ON claude_messages(session_id, timestamp);
CREATE INDEX idx_sessions_project ON claude_sessions(project_path);
CREATE INDEX idx_sessions_role ON claude_sessions(detected_role);
```

**同步引擎工作模式**：

| 场景 | 策略 |
|------|------|
| 首次启动 / 全量重建 | 遍历 `~/.claude/projects/` 所有 `.jsonl` 文件，全量解析入库 |
| 增量同步（定时触发） | 1. 扫描目录发现新文件 → 全量解析新文件<br>2. 对比已有文件 mtime，有变更的从 `last_byte_pos` 继续读取追加 |
| 同步频率 | 默认每 30 秒一次（可配置）；对话视图打开时可手动触发「刷新」 |
| 失败降级 | 同步失败不影响已缓存数据，下次重试；数据库损坏可随时从原始文件重建 |

**会话列表展示字段**：
- 会话标题（首条用户消息前 50 字）
- 检测角色标签（带置信度，高/中/未知三档）
- 开始时间 / 最后活跃时间
- 所属项目路径
- 消息数量（用户/助手/工具）
- 支持按项目筛选、按角色筛选、按时间倒序

**会话列表交互策略**：
- 排序：按 `last_msg_at`（最后活跃时间）倒序
- 首屏加载：50 条，滚动到底部自动加载下一批（前端本地分页，基于已加载全量数据切片）
- 筛选/搜索：标题过滤在前端本地做（不做服务端搜索）
- 角色标签、项目路径、消息数作为辅助识别信息展示

**角色自动识别规则**：
- 优先匹配首条用户消息中的「你是 XX」模式（精确匹配，置信度 1.0）
- 其次通过关键词加权评分（产品/PRD/架构/部署/开发 等关键词）
- 无法识别的标记为 `Unknown`，置信度 0
- 识别结果仅作辅助标签，最终角色以用户映射配置为准

### 6.4 迭代时间轴（只读解析）

- **第一层数据源**：各项目 `docs/progress/INDEX.md` 的「版本列表」表格
- **第二层数据源**：各项目 `docs/progress/iterations/vX.Y.md` 的阶段门禁状态
- **产出内容摘要（三级降级策略）**：
  - **最低保证形态**：阶段名 + 责任角色 + 完成日期 + 产出文档链接（必须展示，不能缺失）
  - **一句话摘要（best-effort）**：能解析到就显示，解析不到就不显示，不报错不阻塞
  - **详细描述**：v0.2 不做，详情走「产出文档链接」和「查看当时的对话」钻取
  - 提取优先级：优先读迭代记录中结构化摘要字段 → 其次从各阶段产出文档标题 + 首段合成 → 最后只展示最低保证形态

### 6.5 生产数据源同步

- **同步方式**：`git fetch` + `git reset --hard origin/main`（镜像语义，不用 pull）
- **实现方式**：cron 或 systemd timer，由 DevOps 在部署阶段落地，默认 5 分钟一次
- **本地变更丢弃策略**：reset 前检测到本地有未提交变更时，记日志并在接入诊断视图标记「已丢弃本地变更」，不阻塞同步
- **失败降级**：某仓库 fetch/reset 失败（网络/权限）时，保留该仓库上次成功的数据继续出快照
- **快照接口新增字段**：每个项目增加 `lastSyncedAt`（最后成功同步时间）和 `syncStatus`（ok / error / local_changes_discarded）
- **接入诊断视图**：展示每个项目的同步状态 + 最后成功同步时间 + 异常原因（如有）+ 本地变更丢弃标记
