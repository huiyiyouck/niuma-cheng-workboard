## 1. 概述

v0.3 是**打磨型 patch 迭代**，设计聚焦五组需求里**有数据模型 / API / 后端流程影响**的部分；纯前端、无架构决策的 US（US-1 视觉、US-7 折叠、US-9 前端错误页）只列现有组件锚点与实现约束，不展开架构设计。

**本设计的三个架构决策（核心）：**

1. **M-1 · 会话角色归属存储模型（首要）**：废弃 v0.2 的 `session_mappings`（1:1）表，改为 `claude_sessions.manual_role` 单列内联；角色归类统一为 `resolved_role = coalesce(manual_role, nullif(detected_role,'Unknown'), 'General')`，天然支持「一角色 N 会话」，并一举兼容 US-4 兜底 General（`Unknown/NULL → General`）。详见 §2.1、§7 取舍、ADR-0001。
2. **US-5 · 迭代归属重建**：新增 `.git` 历史解析器，从各项目 `INDEX.md` 的 git 提交历史重建「当前迭代」随时间变化的**活跃区间**，再用会话活动时间窗匹配，得出「尽力而为」的迭代标签（标注推断、允许中性「未归属」）。这是本迭代后端最大工作量。
3. **API 契约收敛**：`/api/sessions` 成为会话+归类+迭代标签的统一读取入口；打标签用新的 `PUT /api/sessions/role`；US-8 新增 `/api/communications/detail` 按 id 读全文；废弃 `/api/mappings` 三端点（随表废弃）。

**不含架构变更、仅前端/交付的部分**（设计仅登记，实现照 PRD 界面要点 + 现有组件）：US-1 对话视觉（`ChatMessage`/`RoleAvatar`/`Markdown`）、US-6 菜单 5→3（`App.tsx` 视图归位）、US-7 待办折叠（`CrossTodoList` + `overflow-hidden` portal 坑）、US-9 nginx 错误页（DevOps）、US-13 迭代时间轴保留（复用 `IterationTimeline`，仅钻取目标改抽屉）。

**范围红线**（承 PRD）：本迭代 workboard 只写自身本地元数据（`manual_role`），**不回写**被监控项目 / 会话源文件；对被监控项目 `.git` **只读**。

## 2. 数据模型

### 2.1 `claude_sessions`（变更：新增 `manual_role` 列）

在现有表上新增一列（其余列不动）：

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `manual_role` | TEXT | 可空，默认 NULL | 手动打标签（US-10）的角色覆盖值；NULL = 未手动纠正。**workboard 本地元数据，不回写会话源**。取值为工作流角色英文名（`PM`/`Architect`/`Developer`/`DevOps`/`General`）之一 |

已存在、本迭代**继续沿用**的相关列：`detected_role`（`detectRole` 自动识别，识别不出存 `NULL` 或历史遗留 `'Unknown'`）、`source`（`'claude-code'`/`'codex'`，v0.2 已有）、`first_message_at`/`last_message_at`（迭代时间窗匹配用）。

**角色归类规则（贯穿 US-3/4/10/11/12，SQL 层与前端一致）：**

```
resolved_role = coalesce(manual_role, nullif(detected_role, 'Unknown'), 'General')
```

- 优先级：手动标签 > 自动识别（非 Unknown）> General 兜底。
- `nullif(detected_role,'Unknown')` 兼容 v0.2 存量把识别不出写成 `'Unknown'` 的历史数据——**无需数据迁移**即满足 US-4「不再出现 Unknown」。
- 一个会话有且仅有一个 `resolved_role`（US-3「同一会话归属唯一角色」）；「角色 → 会话列表」= 按 `resolved_role` 分组，天然 1:N。

### 2.2 `session_mappings`（废弃 · DROP）

v0.2 的 1:1 手动映射表整表废弃。其唯一有效语义（手动把会话绑到某角色）迁移进 `claude_sessions.manual_role`；表内 `project_id`（workboard 项目 id）为冗余——项目维度天然由 `claude_sessions.project_id`（目录编码 id）+ config `claude_project_id` 映射承载（`handleListSessions` 已实现该映射），迁移时丢弃无损。迁移 SQL 见 §6。

### 2.3 迭代活跃区间 `IterationInterval`（US-5 · 新增，派生模型不落库）

由 `.git` 历史解析器从各项目 `INDEX.md` 重建，**内存派生 + 缓存**（不新增数据库表）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `projectId` | string | workboard 项目 id |
| `version` | string | 迭代版本，如 `v0.2`（来自 INDEX「当前迭代」字段的历史值） |
| `startAt` | string(ISO) | 该「当前迭代=version」首次出现的 commit 时间 |
| `endAt` | string(ISO) \| null | 下一次「当前迭代」变更的 commit 时间；最新区间为 `null`（至今） |

- 一个项目得到按时间排序、首尾相接的区间列表。会话迭代标签 = 用会话时间窗（`first_message_at`~`last_message_at` 的代表点，取 `last_message_at`）落入哪个区间。
- 命中不到（会话早于 INDEX 首 commit、活动时段「当前迭代=无」、非工作流项目、`.git` 不可读降级）→ 标签为 `null`，前端展示中性「未归属·推断不出」。

## 3. API / 接口契约

> 变更 4 项、新增 2 项、废弃 3 项。所有会话读取统一走 `/api/sessions`；打标签与沟通全文各自新增专用端点。

### 3.1 `GET /api/sessions`（变更：返回增强）

| 方法 | 路径 | 入参 | 出参 | 错误码 |
|------|------|------|------|--------|
| GET | `/api/sessions` | `project_id`(可选)、`role`(可选,新增)、`status`(可选)、`limit`/`offset` | `{ items: SessionItem[], total, limit, offset }` | 500 |

- **`SessionItem` 每项新增字段**：`manual_role`(string\|null)、`resolved_role`(string，§2.1 归类结果)、`iteration_label`(string\|null，US-5 迭代标签)、`iteration_inferred`(bool，恒 true，标注「推断」)。`source`(已有) 前端作来源标签。
- **`role` 入参（新增）**：给定则只返回 `resolved_role = role` 的会话（服务端按归类过滤，供 US-3 强限制选择器 / 角色→会话列表）。
- **移除** `status=mapped/unmapped` 对 `session_mappings` 的依赖：`mapped` 重定义为「`manual_role` 或 `detected_role` 非空」，`unmapped` 为「`resolved_role='General'` 且无 manual」（Developer 实现时按此语义，去掉对 `session_mappings` 的 JOIN）。
- 迭代标签在此 handler 内对每个会话按 §2.3 区间匹配计算（区间来自缓存，见 §4.3）。

### 3.2 `PUT /api/sessions/role`（新增：打标签 US-10）

| 方法 | 路径 | 入参(JSON body) | 出参 | 错误码 |
|------|------|--------------|------|--------|
| PUT | `/api/sessions/role` | `{ session_id, role }` | `{ ok:true, session: SessionItem }` | 400(缺参/role 非法枚举)、404(会话不存在) |

- 语义：`UPDATE claude_sessions SET manual_role = $role WHERE id = $session_id`。`role` 必须 ∈ `{PM,Architect,Developer,DevOps,General}`。
- 覆盖自动识别（US-10「覆盖」）；持久化（刷新保留）；不回写会话源。

### 3.3 `DELETE /api/sessions/role`（新增：取消手动标签）

| 方法 | 路径 | 入参 | 出参 | 错误码 |
|------|------|------|------|--------|
| DELETE | `/api/sessions/role?session_id=` | `session_id`(query) | `{ ok:true, session: SessionItem }` | 400、404 |

- 语义：`SET manual_role = NULL`，`resolved_role` 回落到 `detected_role`/General。（前端「撤销纠错」用；非必做 UI，但契约留出。）

### 3.4 `GET /api/communications/detail`（新增：US-8 沟通全文）

| 方法 | 路径 | 入参 | 出参 | 错误码 |
|------|------|------|------|--------|
| GET | `/api/communications/detail` | `id`(query，= 沟通条目 id / 文件名去 `.md`) | `{ id, reqId, content }` | 400(缺 id)、404(文档未找到) |

- 后端复用 `coordination.js` 现有 `readCommunications` 的**已读全文**能力（`readFile` 已在，仅需按 id 定位单文件并返回 `content` 全文，不再只留 `firstHeading`）。
- **出参不含 `sourcePath`（DEV-L2）**：绝对路径属服务器内部信息，不透前端；前端只需 `content` 渲染。
- 路径映射（PRD §6 开放项）：`commItems` 的 id 即 `communications/` 下去扩展名的文件名；按精确 id 定位；找不到走 404「沟通文档未找到」（前端展示 PRD US-8 验收的提示）。
- 只读，无写接口。

### 3.5 废弃：`/api/mappings` GET / POST / DELETE

随 `session_mappings` 表废弃，三端点移除。前端 `MappingDialog` / `SessionSelect` 的写路径改指 §3.2/§3.3；读「角色→会话」改用 §3.1 的 `role` 过滤。（Developer 确认前端无其他引用后移除，避免死端点。）

### 3.6 不变：`/api/snapshot`、`/api/sessions/details`、`/api/timeline/versions`、`/api/timeline/detail`、`/api/sync`、`/api/health`

- `/api/sessions/details`（对话正文）仅需去掉对 `session_mappings` 的 JOIN、补 `resolved_role`，返回结构不破坏。
- `/api/timeline/*`（US-13 时间轴）**契约不变**——US-13 是保留 v0.2 功能，前端复用 `IterationTimeline`，仅前端把阶段「查看对话」钻取目标从全屏 `ConversationView` 改为打开抽屉（§5）。**注意：US-13 时间轴数据源（INDEX 版本列表 + vX.Y 阶段解析）与 US-5 会话迭代标签（INDEX 的 git 历史区间）是两套不同数据源，勿混用。**

## 4. 核心流程

### 4.1 会话归类（贯穿 US-3/4/10/12）
1. sync（`claude-sync.js`，含 Claude + Codex）入库时写 `detected_role`（`detectRole` 结果，识别不出为 `NULL`/`'Unknown'`）与 `source`。**本迭代 sync 管道不改**（US-12 后端已落地）。
2. 读取侧统一按 §2.1 `resolved_role` 归类。Codex 会话（`source='codex'`）与 Claude 会话走**同一** `detectRole` 与归类逻辑（US-12「同一匹配闭环」），无分支。
3. US-4 兜底由 `coalesce(...,'General')` 在读取层完成，`detectRole` 源头可不改（可选增强：扫 assistant 消息进一步降 Unknown，PRD 列为非必做，本设计不纳入必做项，Developer 实现阶段自行评估）。

### 4.2 打标签纠错（US-10）
前端拖拽/⌄菜单选目标角色 → `PUT /api/sessions/role{session_id,role}` → `manual_role` 更新 → 前端乐观更新该会话 `resolved_role` 并重分组（或重拉 `/api/sessions`）。触屏兜底走同一端点（长按/右键菜单）。

### 4.3 迭代重建与标签（US-5，后端大头）
1. **区间重建**：对每个工作流项目，新增解析器执行 `git -C <projectRepo> log --reverse --format=%H|%cI -- docs/progress/INDEX.md`，逐 commit 读出该版本 INDEX 的「当前迭代」字段值（`git show <sha>:docs/progress/INDEX.md` 抽取「当前迭代：vX.Y」），把「当前迭代」值发生变化的点切成 §2.3 区间列表。
2. **缓存**：区间重建有 `git` 子进程开销，**按项目缓存**（随 `/api/sync` 刷新或带 TTL 惰性重建；key = 项目 repo 路径 + INDEX 最新 commit sha）。`handleListSessions` 从缓存取区间，不每请求重跑 git。
3. **匹配**：会话 `last_message_at` 落入某区间 → `iteration_label=version`；落空 → `null`。命中率天花板（基线 62.8%）与「一会话跨多迭代」为已知局限（PRD §6），标签恒标 `iteration_inferred=true`。
4. **生产 .git 降级**（§6）：`git` 调用失败（仓不可读）→ 该项目所有会话 `iteration_label=null`，前端统一「未归属」，功能其余部分不受影响（不抛错、不阻塞会话列表）。

### 4.4 沟通全文（US-8）
需求池「沟通」子 tab 点条目 → `GET /api/communications/detail?id=` → 后端定位单文件读全文返回 `content` → 前端抽屉 `Markdown` 只读渲染；404 走「沟通文档未找到」提示。

### 4.5 对话查看器 当前/历史（US-11，前端状态）
- 后端只提供「角色 → 会话列表」（§3.1 `role` 过滤或前端对全量分组），**不存当前会话、无当前会话 API**（承 PRD §5 与 Architect R1 结论）。
- 前端 `currentByRole: Record<role, sessionId>` 为**临时 state**：某角色默认 current = 其会话列表中 `last_message_at` 最新者；点任一会话 `setCurrent(role, sessionId)`（选择即当前）；刷新/切项目丢失、回落默认。历史会话收进**下拉框**（原型图 2026-07-14 定稿形态，优先于 PRD §3A 线框的平铺列表）。

## 6. 迁移和兼容性

### 6.0 前置：迁移引擎 dollar-quoted 块修复（回环订正 · 实现阶段第一步）

> **回环来源**：实现阶段 Developer 搭全新库 `workboard_dev` 时，`applyMigrations` 跑 `001_init.sql` 报 `42601` 语法错，证伪了下文原「全新库 001→002 链」的**隐含前提**（假定全新库 001 能跑起来）。经 Architect 核实根因确认，本小节订正。

**根因（已读码核实）**：`migrations.js` 的 `splitSqlStatements`（`:100-143`）自己按 `;` 拆语句、只跟踪单引号/双引号/`--` 注释，**不识别 PostgreSQL dollar-quoted 块（`$$…$$`）**。`001_init.sql:56-60` 的 `DO $$ BEGIN IF NOT EXISTS(...) THEN ALTER TABLE ...; END IF; END $$;` 内部有多个 `;`，被误拆成 `DO $$ …ADD CONSTRAINT…UNIQUE(project_id, role)` / `END IF` / `END $$` 三个碎片，逐个 `client.query` 必然 `42601`。**全新库执行 `001` 直接失败、`applyMigrations` 中断，`002` 根本轮不到。**

**为何生产没暴露**：生产 `workboard` 库 `schema_migrations` 已记 `version 1`（历史写入），`applyMigrations` 跳过 `001`，故 bug 仅在**全新库/重建库**（`workboard_dev`）暴露。

**订正方案（Architect 定 · 方案 B：退役手搓拆分器，交 PG 解析）**：
- **`executeMigration` 改为把整个 `.sql` 文件作为单条 `client.query(sql)` 执行**（node-postgres simple query protocol 支持一次执行以 `;` 分隔的多语句，由 **PostgreSQL 服务端**正确解析 dollar-quote / 注释 / 字符串——唯一权威 SQL 解析器）；**退役 `splitSqlStatements`**（不再在 JS 里重实现 SQL 词法分析，从根上消除本类 bug）。
- **保留外层显式事务**：`BEGIN` → `client.query(整文件)` → `INSERT schema_migrations` → `COMMIT`（失败 `ROLLBACK`），维持 DevOps 已认可的 `002` 原子性（`ALTER`+`UPDATE`+`DROP` 不可拆）。
- **约束**：simple query（多语句）不支持参数占位符（`$1/$2`），故**迁移 `.sql` 文件须为纯字面量 SQL、不用参数**（现有 `001`/`002` 均满足；约定后续迁移文件遵守。`INSERT schema_migrations` 那条是文件外的独立参数化 query，不受影响）。
- **不改 `001_init.sql` 内容**（迁移 append-only 不可变），只改**执行它的引擎**——修复后 `001` 的 DO 块由 PG 正确执行。
- **为什么不选方案 A（给 `splitSqlStatements` 打补丁支持 `$$`）**：继续手搓拆分器还要处理 `$tag$`、嵌套 dollar-quote、字符串内 `$$`、注释内 `;` 等边界，脆弱且是本 bug 的同源隐患；方案 B 代码更少、更健壮，顺带根治 DEV-M1 提到的引擎层技术债。

**修复后两条路径的正确性（订正原「全新库收口」表述）**：
- **全新库（`workboard_dev`）**：引擎修复后 `001`（建 `claude_sessions`/`claude_messages`/`session_mappings` + DO 块约束）一次执行成功 → `002`（`ALTER` 加 `manual_role`；`UPDATE … FROM session_mappings` 迁移 **0 行**，空表；`DROP` 空表）成功。链**成立**（此前不成立是因 `001` 跑不起来，非逻辑错）。
- **生产库（`workboard`）**：`schema_migrations=v1`、`001` 跳过；`002` **不含 DO 块**（纯 `ALTER/UPDATE/DROP`），新旧引擎均正确执行；生产 `session_mappings` **6 行真实手动映射**由步骤 2 `UPDATE` 迁进 `manual_role`（不丢数据），再 `DROP`。引擎修复对生产 `002` 无害（无 dollar-quote），并叠加 OPS-1 的 `pg_dump` 备份兜底。

**归属**：迁移引擎修复是 `002` 可靠落地的**前提**，与 M-1 存储改造同属实现阶段第一批工作、须在跑 `002` 之前落地；Developer 在 `workboard_dev` 全新库实测 `001→002` 通过即验证生效。

### 6.1 迁移脚本与 DDL 清理

**新增迁移 `src/server/migrations/002_session_role_model.sql`**（走已有 `migrations.js` 版本化机制，`schema_migrations` 记录；**依赖 §6.0 引擎修复方能在全新库可靠执行**）：

```sql
-- 1) 新增 manual_role 列
ALTER TABLE claude_sessions ADD COLUMN IF NOT EXISTS manual_role TEXT;

-- 2) 迁移 v0.2 手动映射：session_mappings.role → claude_sessions.manual_role
--    （幂等：仅当存量表存在时执行；project_id 冗余丢弃，角色维由 session 承载）
UPDATE claude_sessions s
   SET manual_role = m.role
  FROM session_mappings m
 WHERE s.id = m.session_id
   AND s.manual_role IS NULL;

-- 3) 废弃 1:1 映射表
DROP TABLE IF EXISTS session_mappings;
```

- **同时清理 DDL（DEV-M1 订正）**：本项目现有**两套重复的 schema 真源**——① `migrations/001_init.sql`（`applyMigrations` 跑，版本化、`schema_migrations` 记录已跑不重跑，由每个 API handler 首次请求触发）；② `db.js` `runSchemaDDL`（`ensureSchema()` 触发，**非版本化**、进程内首次 sync 时执行一次 `db.js:71` `_schemaReady`，由 `claude-sync.js:438/497` 调）。**两者都建 `session_mappings`**。清理须两处协同：
  - **`runSchemaDDL` 删 `session_mappings` 的 `CREATE TABLE`/约束/索引段（`db.js:125-145`）**——否则进程重启后首次 sync 又 `CREATE TABLE IF NOT EXISTS session_mappings` 把空表建回、绕过 `002` 的 DROP（即「DROP 后复活空表」，OPS 交叉印证的坑）。
  - **`runSchemaDDL` 无须补 `manual_role`**（采纳 DEV-M1 建议③）：`manual_role` 由 `002` migration 建列（版本化，全新库 001→002、存量库 002 补齐），且读 `manual_role` 的 handler 均先调 `applyMigrations`；sync 不写 `manual_role`、缺列不影响 sync。故冷启动一致由 migrations 保证，不靠 `runSchemaDDL`（原设计「补入 runSchemaDDL」基于「它是冷启动读路径」的同一误判，撤回）。
  - **`001_init.sql` 不改**（迁移 append-only 不可变、存量库已记录不再跑）——全新库靠 `002` 的 `DROP TABLE IF EXISTS` 在 `001` 之后收口（`001` CREATE → `002` DROP，最终无表）。**前提：§6.0 引擎修复后 `001` 方能在全新库跑起来**（否则 `001` 42601 中断、`002` 轮不到）。
  - 触发点修正：`runSchemaDDL` 非「每次启动」而是「进程内首次 sync 触发一次」。**可选根治（超出 v0.3 范围，记未来项）**：让 `claude-sync` 改走 `applyMigrations`、退役 `runSchemaDDL` 内联 DDL，消除两套真源；本迭代只做上面的最小闭环。
- **US-4 存量兼容**：无需数据迁移，`nullif(detected_role,'Unknown')` 在读取层兜底。
- **`detected_role`/`source`/messages 数据**：全部保留，不受影响。
- **生产 `.git` 可读性（DevOps 重点，开放项）**：US-5 区间重建要求生产部署形态下能 `git -C <各项目仓>` 只读执行。DevOps R1 已核实生产被监控项目仓 `/root/Project/niuma-cheng-*` 为真实克隆、root 可读；不可读则 §4.3 降级（迭代标签全 `null`），不阻塞其余功能。**`git` 调用须限定各被监控项目仓根、只读子命令（`log`/`show`），不触碰工作区、不写。**
  - **OPS-2 生产自监控 .git 陷阱**：workboard 自身若在 config 里把 path 指向部署目标 `/opt/workboard-prod/app`（`cp -a` 产物、**非 git 目录**），则 US-5 对 workboard 自己恒 `null`（静默漏测）——各项目 path 须指真实 git 克隆，部署就绪检查核对。
- **`DROP TABLE session_mappings` 生产执行风险（DevOps 重点）**：`DROP` 不可逆，但数据已在步骤 2 迁移进 `manual_role`；DevOps R1 已确认迁移在**单事务**内原子执行（`ALTER`+`UPDATE`+`DROP` 整体 `BEGIN…COMMIT`，无半迁移态），且已给回滚策略（restart 前 `pg_dump session_mappings` + 健康检查触发回滚 + 反向 cp 上一版）。
  - **OPS-1 备份闸口**：`applyMigrations` 惰性自动应用（每 handler 首次请求触发，含 `/api/health`），`systemctl restart` 后第一笔流量即自动 `DROP`、无人工闸口——故 **`pg_dump session_mappings` 须定为部署 runbook 里 `restart` 之前的强制步骤**（落部署就绪检查清单）。迁移后跑一次 sync 复查 `session_mappings` 仍不存在（验证 DDL 清理生效）。

## 7. 风险和取舍

- **M-1 存储模型 路径 A vs B**（详见 ADR-0001）：选路径 A（`manual_role` 落列 + 废表）。替代路径 B（保留 `session_mappings` 放宽到 1:N）被否——两套归属来源并存、归类需 JOIN 两表、`project_id` 冗余，复杂且与「会话行承载一切归属」的干净模型相悖。路径 A 唯一代价是 `DROP TABLE`（不可逆），用迁移前备份 + 步骤 2 数据迁移覆盖。
- **US-5 迭代重建性能**：`git log`/`git show` 子进程按项目缓存（§4.3），避免每请求重跑；INDEX 提交历史通常几十条量级，单项目重建成本可接受。风险在缓存失效策略（TTL vs sync 触发），Developer 实现阶段定。**DEV-L1 实现注意**：每 commit 一次 `git show` 首次重建偏慢，可用 `git log -p -- docs/progress/INDEX.md` 一次拿到全历史 diff 解析「当前迭代」变化，减少子进程数。
- **迭代标签「尽力而为」**（BCR-013 局限）：一会话未必对应单一迭代，标签恒标「推断」、留中性「未归属」；不追求全覆盖（命中率天花板 62.8% 已知）。
- **生产 `.git` 不可读**：唯一可能致 US-5 整体降级的外部依赖，已交 DevOps 部署检查确认 + §4.3 降级兜底。
- **菜单 5→3 量级**（PRD §6）：C 组是唯一信息架构重构、量级最大，与 A/B 主题不同，实现最后做、可单独砍，不影响 A/B 主线。
- **`detectRole` 扫 assistant 消息**：本设计**不纳入必做**（PRD 定位可选增强）；兜底 General 由归类层完成，已满足 US-4 必做验收。
- **迁移引擎手搓 SQL 拆分器（回环订正）**：`splitSqlStatements` 自己拆语句是本类 bug 温床（dollar-quote 未识别致全新库 001 失败）。选方案 B（交 PG 服务端解析、退役拆分器）而非方案 A（补丁支持 `$$`）——B 代码更少、边界更全、根治同源隐患，唯一约束是迁移文件不用参数占位符（已约定）。见 §6.0。
