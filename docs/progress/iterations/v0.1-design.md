## 1. 概述

v0.1 采用“本地 Node 只读聚合服务 + 复用 `frontend/` React 原型”的结构。

核心原则：

- **只读**：服务只读取 `projects.config.json`、各项目 `docs/progress/` 和 coordination 真源，不写回任何被监控项目。
- **配置驱动**：项目清单、路径、类型、启用状态和可选线上地址都来自 `projects.config.json`。
- **项目级隔离**：单项目配置异常、路径异常或读取异常只影响该项目卡片/诊断行，不阻断其他项目。
- **区域级降级**：coordination 真源不可读时，只让跨项目区域显示降级提示，不让工作台整页崩溃。
- **稳定接口**：后端把 Markdown 和 config 解析成稳定 JSON，前端只消费模型和错误枚举，不直接解析 Markdown。

建议目录：

```text
.
├── package.json
├── projects.config.json
├── src/
│   ├── server/
│   │   ├── index.js
│   │   ├── config.js
│   │   ├── snapshot.js
│   │   ├── errors.js
│   │   └── parsers/
│   │       ├── markdown-table.js
│   │       ├── project-index.js
│   │       ├── roles.js
│   │       ├── coordination.js
│   │       └── workflow-source.js
│   └── shared/
│       └── schema.js
└── frontend/
```

根目录 `package.json` 负责本地启动。`frontend/` 保留 Vite + React 原型基线，后续由 Developer 将 mock 数据替换为 API 数据。

## 2. 数据模型

### 2.1 配置模型

```ts
type ProjectKind = "workflow-source" | "business" | "coordination" | "workboard";

type ProjectConfig = {
  id: string;
  name: string;
  path: string;
  kind: ProjectKind;
  enabled?: boolean;
  url?: string;
};

type AppConfig = {
  refresh?: {
    interval_seconds?: number;
  };
  projects: ProjectConfig[];
  coordination?: {
    project_id?: string;
  };
};
```

校验规则：

- `projects` 必须是非空数组。
- `id` 必须唯一，建议只允许 `[a-z0-9_-]+`。
- `kind` 必须属于四类枚举。
- `path` 支持相对路径、绝对路径、`${PROJECT_HOME}` 展开。
- `enabled` 缺省视为 `true`。
- `url` 可选；缺失不构成异常，部署页灰显“未配置线上地址”。
- `refresh.interval_seconds` 缺省为 `60`，小于 `5` 或非数字时回退 `60` 并给诊断 warning。
- `coordination.project_id` 若存在，必须指向 `kind = coordination` 的项目；缺失时自动取第一个 coordination 项目。

### 2.2 状态枚举

接口状态统一使用英文枚举，前端映射中文展示。

```ts
type IntegrationStatus =
  | "integrated"
  | "not-bootstrapped"
  | "config-error"
  | "read-error"
  | "disabled";

type ReadSeverity = "info" | "warning" | "error";

type ProjectMode = "标准迭代" | "非迭代" | "未选择" | null;
```

完成态判断：

- 跨任务待办状态包含 `已完成`、`完成`、`已关闭`、`已归档`、`✅` 时视为完成，不进入待办汇总。
- 状态为空、`待启动`、`待评估`、`进行中`、`阻塞`、`Review中`、`未收尾` 均视为未完成。
- 解析不到状态列时，该表进入 `read-error` 降级，不猜测待办完成态。

- 中文项目名、配置 `name`、常用短名可共同作为 alias。v0.1 先内置 `id` 和 `name`，不额外引入 alias 配置。

## 3. API / 接口契约

### 3.1 `GET /api/snapshot`

返回整批看板快照，是前端 60s 轮询的主接口。

响应：

```json
{
  "generatedAt": "2026-06-23T10:00:00.000Z",
  "refreshIntervalSeconds": 60,
  "summary": {
    "activeProjects": 3,
    "blockedProjects": 1,
    "abnormalProjects": 0,
    "openTodos": 4
  },
  "projects": [],
  "diagnostics": [],
  "crossProject": {
    "status": "ok",
    "activeRequestCount": 1,
    "blockerCount": 0,
    "contractCount": 1,
    "communicationCount": 1,
    "bcrCount": 2,
    "highlightRequests": [],
    "blockers": [],
    "requests": [],
    "bcrs": [],
    "communications": [],
    "errors": []
  },
  "todos": [],
  "errors": []
}
```

HTTP 状态：

- `200`：成功生成快照；即使局部项目读取失败，也返回 `200`，错误写入对应项目/区域。
- `500`：根配置文件不可读、JSON 语法错误且无法生成任何快照。前端显示整页读取异常。

### 3.2 `GET /api/config/diagnostics`

返回配置与文件检查。可由接入诊断视图单独刷新，也可直接复用 `/api/snapshot.diagnostics`。

v0.1 可只实现 `/api/snapshot`；若实现单独接口，其响应类型为 `ProjectDiagnostics[]`。

### 3.3 静态资源

生产/本地预览模式：

- Node 服务读取 `frontend/dist` 并提供静态资源。
- 未命中 `/api/*` 的路径回退到 `frontend/dist/index.html`，支持前端单页应用。

开发模式：

- `frontend` 使用 Vite dev server。
- Vite 代理 `/api` 到 Node 后端，或前端通过 `VITE_API_BASE` 指向后端。

## 4. 核心流程

### 4.1 启动流程

1. Node 服务从当前工作目录读取 `projects.config.json`。
2. 校验 config schema，生成配置级 diagnostics。
3. 启动 HTTP 服务。
4. 每次请求 `/api/snapshot` 时现场读取文件并聚合；v0.1 不做持久缓存。
5. 后续如文件数量增长，可增加进程内短缓存，缓存时间不得超过 `refresh.interval_seconds`。

### 4.2 路径解析

解析顺序：

1. 若 `path` 以 `/` 开头，按绝对路径处理。
2. 若包含 `${PROJECT_HOME}`，用环境变量展开；环境变量缺失则该项目 `config-error`。
3. 其他相对路径以 `projects.config.json` 所在目录为基准。
4. 解析后用 `fs.realpath` 获取实际路径；不存在则 `config-error`。

### 4.3 按 kind 解析

`business` / `workboard`：

- 必需：`docs/progress/INDEX.md`
- 可选：`docs/progress/roles/`
- 缺 `INDEX.md`：`not-bootstrapped`
- `INDEX.md` 存在但不可读或解析失败：`read-error`
- 提取当前迭代、当前模式、当前阶段、阻塞项、下一步入口、跨任务待办。

`coordination`：

- 必需：`REQUESTS.md`、`STATUS.md`
- 可选：`contracts/`、`communications/`
- 不要求 `docs/progress/INDEX.md`。
- 不进入工作台项目卡网格；进入接入诊断，并作为跨项目协作横条和跨项目视图的数据源。

`workflow-source`：

- 不要求 `docs/progress/INDEX.md`。
- v0.1 检查仓库目录存在即可视为接入；若存在 `docs/ROADMAP.md`、`docs/baseline/`、`docs/templates/`，可生成“工作流真源”摘要。
- 不展示角色看板。

### 4.4 Markdown 解析

统一使用“标题定位 + 表格列名”的解析策略：

- 先按二级/三级标题定位区域，例如 `## 当前项目状态`、`## 跨任务待办`、`## 需求池`、`## 基线修正提案池`。
- 在区域内寻找第一个 Markdown 表格。
- 用表头列名映射字段，不依赖固定列序。
- 表格解析失败时返回区域级错误；能保留原文片段时，放入 `error.rawExcerpt` 供诊断抽屉展示。
- 不用正则跨全文硬抓字段，避免命中 Review 历史或归档内容。

`INDEX.md` 当前状态字段不是表格，按列表项解析：

- `- 当前迭代：`
- `- 当前模式：`
- `- 当前阶段：`
- `- 阻塞项：`
- `- 下一步入口：`

### 4.5 角色日志解析

角色文件优先级：

1. `{role}-current.md`
2. `{role}-summary.md`
3. `{role}.md`
4. `{role}-archive.md` 不默认读取，除非前三者都不存在且文件很小

v0.1 只提取最近动作和下一步：

- 优先找最近一个 `## YYYY-MM-DD` 段落。
- 在段落内读取 `结论`、`下一步入口`、`遗留问题/风险`。
- 解析失败时角色状态为 `read-error`，不影响项目主卡。

### 4.6 跨项目解析

`REQUESTS.md`：

- `## 需求池` 表格解析为 `CrossProjectRequest[]`。
- `## 基线修正提案池` 表格解析为 `BcrItem[]`。
- 需求筛选：`已关闭` 为关闭；其他状态属于进行中。
- BCR 筛选：`已回流下游`、`已拒绝`、`转 v2 候选` 为终态；其他属于进行中。

`STATUS.md`：

- `## 跨项目阻塞与谁等谁` 下的小节为阻塞/依赖条目。
- 含 `谁等谁：无` 或 `状态：已完成` 的条目不计入活跃阻塞。
- v0.1 不解析已解除历史阻塞列表。

`contracts/` 与 `communications/`：

- 计数按 `*.md` 文件数量，排除 `README.md`。
- 沟通摘要从文件首个标题、`摘要` 字段或前 300 字中提取。

### 4.7 轮询与错误隔离

前端：

- 启动后立即请求 `/api/snapshot`。
- 根据 `refreshIntervalSeconds` 启动轮询。
- 用户打开 Drawer 时，轮询不关闭；更新后用 `id` 保留当前选中对象。
- 若选中对象消失，Drawer 展示“对象已不存在或被禁用”降级状态。
- 若选中对象变为 `read-error`，Drawer 展示该对象错误摘要和诊断信息。

后端：

- 每次快照生成独立错误列表。
- 项目级错误写入该项目 `diagnostics.errors` 和 `ProjectSummary.errorSummary`。
- coordination 错误写入 `crossProject.errors`。
- 根配置不可读才返回 HTTP 500。

## 5. 前端或交互影响

前端现状 `frontend/src/app/App.tsx` 仍包含 mock 数据。实现阶段替换路径：

1. 新增前端数据类型文件，与后端返回模型保持一致。
2. 把 `PROJECTS`、`CROSS_TODOS`、`CROSS_PROJECT_ITEMS`、`BCR_ITEMS`、`COMM_ITEMS` 替换为 `useSnapshot()` hook。
3. 保留现有 Sidebar、Tabs、Drawer、Card、Badge、Table 结构。
4. 工作台项目卡过滤 `kind !== "coordination"`。
5. 接入诊断表展示所有配置项目，包括 `coordination`。
6. 部署页仅当 `url` 非空时可点击；否则灰显。
7. 所有空态和错误态都从后端枚举驱动，不在前端猜测。

建议前端状态：

```ts
type SnapshotUiState =

  | { state: "ready"; data: SnapshotResult }

type DrawerState =
  | { kind: "closed" }

```

## 6. 迁移和兼容性

### 6.1 工程脚本

根目录新增 `package.json`，建议脚本：

```json
{
  "scripts": {
    "dev": "node src/server/index.js --dev",
    "build": "npm --prefix frontend run build",
    "start": "node src/server/index.js"
  }
}
```

`npm start`：

- 启动本地 Node 服务。
- 如果 `frontend/dist` 存在，直接服务静态产物。
- 如果 `frontend/dist` 不存在，返回明确提示：先执行 `npm run build`。

`npm run dev`：

- 后端监听固定端口，例如 `5174`。
- 前端 Vite 仍用 `frontend` 的 `npm run dev` 或由 Developer 后续选择并行脚本。

### 6.2 配置兼容

历史 `projects.config.json` 不含 `url` 字段，必须继续有效。Owner 已确认当前配置补入 xiaobao URL：

```json
{
  "id": "xiaobao",
  "url": "https://news.huiyiyou.cloud"
}
```

未配置 `url` 的项目仍正常显示，部署页灰显“未配置线上地址”，不计为配置异常。

### 6.3 Markdown 兼容

各项目 `INDEX.md` 的表格可能有历史内容和自由文本。v0.1 只保证解析当前约定区域：

- `当前项目状态`
- `跨任务待办`

若某项目还未接入该结构：

- 有 `INDEX.md` 但缺区域：项目保持 `integrated`，对应字段为 `null`，diagnostics 给 warning。
- `跨任务待办` 缺失：视为空待办，不报错。

## 7. 风险和取舍

| 风险 | 严重度 | 处理 |
|------|--------|------|
| Markdown 表格列名漂移导致解析失败 | 中 | 表头映射 + 区域级错误 + 原文片段兜底 |
| `ai` 短 id 误匹配 | 中 | 使用边界匹配，优先项目名和显式字段，不做简单 substring |
| 角色日志形态不一致 | 中 | 按 current/summary/role.md 优先级解析，失败只影响角色卡 |
| `url` 字段缺失 | 低 | schema 设为可选，部署页灰显 |
| 60s 全量读取性能 | 低 | v0.1 项目少，现场读取可接受；后续再加进程内缓存 |
| 前端 peerDependencies 中 React 可选 | 低 | Developer 实现阶段整理为可稳定安装/构建的依赖 |

取舍：

- 不引入数据库：降低复杂度，符合只读看板定位。
- 不做文件监听/WebSocket：60s 轮询足够满足 Owner 监督场景。
- 不在前端解析 Markdown：避免 UI 和数据真源格式耦合。
- 不把 coordination 当普通项目卡：符合已定稿 UI 数据规格，避免无迭代字段造成误读。

  - 设计已明确 v0.1 采用本地 Node 只读聚合服务 + `frontend/` React 原型基线，工程路线和 PRD 范围一致。
  - `/api/snapshot` 的快照模型、HTTP 状态、项目级错误隔离和 coordination 区域级降级均已定义，能支撑 60s 轮询和单项目失败不阻断整页。
  - `projects.config.json` schema 已覆盖 `id` 唯一、`kind` 枚举、路径解析、`${PROJECT_HOME}`、可选 `url`、`refresh.interval_seconds` 回退和 coordination 项目定位。
  - Markdown 解析策略采用“标题定位 + 表格列名”，并明确解析失败返回区域级错误和原文片段，避免前端直接解析 Markdown。
  - 前端替换路径清楚：从 `App.tsx` 内置 mock 迁移为 `useSnapshot()`，保留既有 Sidebar / Tabs / Drawer / Card / Table 结构。
- 问题：
  - 严重度：中。根目录脚本只给出建议形态，`npm start` 在 `frontend/dist` 不存在时要求先 `npm run build`；实现阶段需明确首次启动体验，避免 Owner 执行 `npm start` 直接看到不可用页面。
  - 严重度：中。设计把 Markdown 表格解析失败定义为区域级错误，但跨项目 `REQUESTS.md` / `STATUS.md` 的实际标题和表格列名可能继续演化；实现阶段应把解析器测试样例直接覆盖当前真实文件，不能只测人工构造样例。
  - 严重度：低。`frontend/package.json` 当前 React / ReactDOM 在 optional peerDependencies；实现阶段需要整理依赖，保证新环境可稳定安装、构建和运行。
- 建议：
  - 实现第一步先补根目录 `package.json` 与最小 Node 服务，再接入 `frontend/dist` 静态服务，形成可运行闭环后再扩展解析器。
  - 为 `config`、Markdown table、project index、coordination、roles 解析器建立集中单元测试，测试 fixture 优先取当前仓和真实项目文件的最小脱敏片段。
  - 前端接入时先做 `SnapshotUiState` 与 `DrawerState`，用项目 `id` / 需求 `id` 保留打开中的 Drawer，避免轮询刷新造成选中对象丢失。

### R1 — Tester Review

- 结论：通过，带中风险进入实现阶段处理。
- Review 范围：异常降级、状态枚举、跨项目待办和 coordination 数据解析是否可验证。
- 依据：
  - 设计已把项目级读取失败、coordination 区域读取失败和根配置不可读分成不同降级层级；可分别覆盖“单项目失败不影响整页”“跨项目区降级不崩页”“根配置失败返回整页错误”三类验收。
  - `IntegrationStatus`、`ProjectMode`、`ReadSeverity`、`ProjectKind` 和前端 `SnapshotUiState` 已明确，能支撑 PRD 中「未接入 / 配置异常 / 读取异常 / 已禁用」等状态断言。
  - 跨任务待办完成态白名单、缺状态列降级为 `read-error`、`coordination` 不进入项目卡网格、`workflow-source` / `coordination` 不因缺 `INDEX.md` 异常，均已形成可测规则。
  - coordination 解析已按 `REQUESTS.md`、`STATUS.md`、`contracts/`、`communications/` 分源定义，且与当前真实文件的「需求池」「基线修正提案池」「跨项目阻塞与谁等谁」标题结构一致。
- 问题：
  - 严重度：中。`SnapshotError` 被多个模型引用，但字段结构尚未定义；实现阶段需至少固定 `code`、`message`、`severity`、`sourcePath`、`rawExcerpt` 或等价字段，否则错误摘要、诊断抽屉和解析失败测试只能按实现细节补猜。
  - 严重度：中。配置 schema 已要求 `id` 唯一，但设计只明确 JSON 语法错误返回 HTTP 500，未明确重复 `id`、空 `projects`、`coordination.project_id` 指向错误时是整批失败还是配置级降级；实现阶段需把这些非法配置各自写成可断言用例。
  - 严重度：低。`STATUS.md` 的阻塞解析规则依赖小节内自由文本字段；实现阶段需要用当前真实 `STATUS.md` fixture 覆盖“谁等谁：无”“状态：已完成”“v1 生效中但无阻塞”三种非活跃阻塞，避免误计入 `blockerCount`。
- 建议：
  - 单元测试按解析器分层建立：config、路径解析、Markdown table、project index、roles、coordination，每类至少覆盖正常、缺文件、不可读/解析失败、字段缺失四类。
  - 集成测试固定一组最小 fixture：一个正常 business、一个缺 `INDEX.md` 的 business、一个 workflow-source、一个 coordination、一个 read-error 项目，用 `/api/snapshot` 断言整批快照不被局部错误打断。
  - 前端测试重点断言状态映射和降级展示，不直接复测 Markdown 解析细节；后端解析器负责保证枚举和错误对象稳定。
