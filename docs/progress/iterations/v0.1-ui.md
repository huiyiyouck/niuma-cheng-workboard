## 1. 用户流程

单用户（Owner）只读监督场景：

```
打开看板 → 默认【工作台】：项目卡片网格 + 待办汇总
   │
   ├─ 点项目卡 → 右抽屉滑出该项目详情（迭代/角色/待办/入口）→ 关闭回总览
   ├─ 左菜单切【部署】 → 点项目 → 新标签打开线上地址
   ├─ 左菜单切【接入诊断】 → 点某行 → 右抽屉看接入详情/错误摘要
   └─ 左菜单切【跨项目】 → Tabs 看 需求池 / 谁等谁 / 沟通
```

## 3. 关键交互状态

每个视图需覆盖：**加载中 / 空数据 / 读取异常 / 正常**。

- **项目卡**：加载骨架；单卡读取失败 → 显示「读取异常」徽标 + 错误摘要，**不阻断其他卡渲染**；`business`/`workboard` 无 `INDEX.md` → 「未接入团队工作流」徽标。
- **抽屉**：打开/关闭动画；内容加载态 / 错误态。
- **诊断表**：`配置异常` / `读取异常` 行高亮，可点开抽屉看错误摘要。
- **跨项目**：需求池空 → 空态提示；coordination 仓不可读 → 区域级降级提示。
- **部署**：无 `url` 配置的项目 → 灰显「未配置线上地址」。

## 4. 视觉约束

- **风格基线**：`docs/knowledge/ui/design-system.md`（shadcn/ui 浅色体系）。
- **前端技术栈**：React 18 + Tailwind v4 + shadcn/ui（Vite 构建），与小报统一，组件与 token 直接复用。〔R1 Owner 定〕
- **关键 token**：背景 `#f6f7f9` / 卡片白 / 主色 `#030213` / 圆角 10px / 字重 ≤500 / lucide 图标。
- **明暗**：浅色为主，深色模式可选。〔R1 Owner 定〕
- **主色**：沿用 `#030213`（与小报统一，不另设强调色）。〔R1 Owner 定〕
- **状态徽标配色**〔R1 Owner 定〕：

  | 接入状态 | 配色 |
  |----------|------|
  | 已接入 | primary / 绿 |
  | 未 Bootstrap | muted 灰 |
  | 配置异常 | destructive 红 |
  | 读取异常 | chart-4 橙 |
  | 已禁用 | 描边灰 |

## 5. 验收标准

1. 当打开看板，应默认展示【工作台】，列出 `projects.config.json` 中所有启用项目的卡片。
2. 当某项目读取失败，应在该卡片显示「读取异常」徽标与错误摘要，且**不影响其他卡片渲染**。
3. 当点击项目卡，应从右侧滑出该项目详情抽屉；关闭后回到原视图与滚动位置。
4. 当 `business`/`workboard` 项目缺少 `INDEX.md`，应显示「未接入团队工作流」徽标（`coordination`/`workflow-source` 不因此判未接入）。
5. 当切换左菜单，中间内容区应切换对应视图，且左菜单高亮当前项。
6. 接入诊断表每行应展示接入状态徽标；`配置异常`/`读取异常` 行可点开抽屉查看错误摘要。
7. 四视图在无数据时应显示空态提示，而非空白或报错。

### R1 — Architect Review

- 结论：通过。
- Review 范围：各项目 `INDEX.md` / `roles/` 与 `coordination` 数据是否支撑四视图字段、接入状态徽标和右抽屉详情。
- 依据：
  - `projects.config.json` 已提供项目 `id` / `name` / `path` / `kind` / `enabled`，可支撑项目枚举、路径解析、接入诊断和按 `kind` 分流。
  - 普通项目的 `docs/progress/INDEX.md` 已包含当前迭代、当前模式、当前阶段、阻塞项、下一步入口等字段，足以支撑工作台项目卡主字段。
  - `docs/progress/roles/` 可作为角色看板入口和最近角色状态来源；缺失角色日志可按“未启用/暂无日志”降级展示。
  - `coordination` 类型项目不要求 `docs/progress/INDEX.md`，应按立项定位读取 `REQUESTS.md`、`STATUS.md`、`contracts/`、`communications/` 等跨项目真源。
- 问题：
  - 严重度：中。UI 方案里的项目卡字段偏普通项目模型，`coordination` / `workflow-source` 这类 `kind` 没有当前迭代、模式、阶段时，需要在设计/实现阶段明确字段降级和替代摘要，避免被误判为读取异常。
  - 严重度：中。接入状态文案同时出现“未 Bootstrap”“未接入团队工作流”，后续接口契约应统一为稳定枚举，再由 UI 映射展示文案和 Badge 配色。
  - 严重度：低。部署视图的 `url` 与在线状态是可选字段；当前配置未提供 `url`，第一版应明确无 `url` 时只灰显，不把在线探测作为必需数据源。
- 建议：
  - 设计阶段定义项目摘要模型，例如 `ProjectSummary` / `ProjectDiagnostics` / `CoordinationSummary`，按 `kind` 明确必填、可选和降级字段。
  - 接入状态建议先收敛为 `connected` / `not_bootstrapped` / `config_error` / `read_error` / `disabled`，再映射中文徽标。
  - 60s 轮询应返回整批读取结果，单项目错误隔离在项目级结果内，避免一个项目读取失败影响整体视图。

### R1 — Developer Review

- 结论：通过，带中风险后置处理。
- Review 范围：shadcn 组件落地成本、左菜单 + 右抽屉布局工程可行性、60s 轮询下的渲染策略。
- 依据：
  - Sidebar / Card / Badge / Drawer / Table / Tabs / Tooltip / Sonner / lucide 的组件组合属于常规中后台界面，交互复杂度可控。
  - 左菜单 + 中内容区 + 右 Drawer 的布局可以在单页应用中稳定实现，抽屉承载项目详情不会破坏总览滚动位置。
  - 四视图均已要求覆盖加载中、空数据、读取异常、正常状态，便于实现阶段做项目级错误隔离和局部降级渲染。
  - 60s 轮询适合用批量读取结果刷新内存状态；只要后端返回项目级结果数组，前端无需因单项目失败重刷整页或阻断其他卡片。
- 问题：
  - 严重度：中。`project-context.md` 当前写明第一版前端为静态 HTML / CSS / JavaScript，但 UI 方案映射到 shadcn/Radix/Tailwind 组件体系；实现前需明确是正式切换为 React + shadcn 脚手架，还是仅按 shadcn 视觉规范做静态实现。
  - 严重度：中。右抽屉详情涉及项目详情、接入详情、需求详情三类内容，建议设计阶段先定义统一 Drawer 状态模型，避免实现时为每个视图堆独立状态导致关闭、加载、错误处理不一致。
  - 严重度：低。60s 轮询需要处理抽屉打开期间的数据刷新策略；建议刷新项目列表时保留当前选中对象，并在对象消失或读取失败时在抽屉内展示降级状态。
- 建议：
  - 若采用 shadcn 真实组件，v0.1 实现阶段应先建立 React + Tailwind + shadcn 基础脚手架，再实现 Node 本地 HTTP 服务；这会改变当前项目背景中的第一版前端技术栈描述。
  - 设计阶段定义 `ProjectSummary`、`ProjectDiagnostics`、`DrawerState` 和跨项目条目摘要模型，并把 Architect 提到的接入状态枚举作为接口稳定字段。
  - 轮询实现采用整批 `/api/snapshot` 类接口返回，前端以项目 `id` 做稳定 key，项目级错误留在对应项目结果内。
