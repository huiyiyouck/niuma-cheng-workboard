## Review 记录

## 1. 迭代目标

打磨「项目管理工作台」v0.2 上线后暴露的多处体验（**不对被监控项目 / 业务数据可写**）：

1. **对话查看器从「单调播放器」升级为「项目内工作台」**：进入某会话的对话后，可在页内横向切换同项目其他角色、直接换/配会话，不必返回上一级；同时改善对话本身的视觉层级与可读性。
2. **会话匹配准确度落到「解析侧正解」**：按 BCR-013 评估结论（不进框架、转 workboard 解析侧），把角色识别与迭代归属做实——角色维改进 `detectRole` 并兜底 `General`，迭代维新增 INDEX-git-history 重建 session↔迭代。
3. **菜单信息架构精简 5→3**（方案 A，独立可砍）：顶级菜单收敛为「看板 / 项目会话 / 需求池」。

另含搭车项：会话匹配的**强限制 + 拖拽纠错闭环**（US-3/10）、**1:N 多会话切换**（US-11）、**多来源会话同步 Codex**（US-12）、需求池**沟通详情**（US-8）、看板**待办折叠 / 长文本**（US-7）、生产**友好错误页**（US-9）。

**本迭代不做**：对被监控项目 / 业务数据的增删改查、对话输入交互、任何 agent-workflow 框架侧改动。（会话映射、手动打标签等 workboard 本地配置 / 元数据写不属此列——不回写被监控项目。）

## 2. 用户故事

### A 组 · 对话查看器三件套（主线）

- **US-1 对话视觉打磨**：作为 Owner，我在查看某会话对话时，想看到层级清晰、可读性好的界面，而不是单调的两色气泡堆叠，以便快速读懂这段对话在讲什么。
- **US-2 页内切角色**：作为 Owner，我进入某项目的对话后，想在**对话抽屉内（左侧角色菜单）**直接切换到该项目的其他角色（如从产品切到开发、架构师），以便对照不同角色的工作，而不用返回重新进入。
- **US-3 页内配会话（1:N）**：作为 Owner，我在对话查看器内为某角色**配置其归属的会话**——一个角色可归属**多个**会话（通过识别 / 打标签归入）；当角色下还没有会话时，页内给我选择入口把会话归入。
- **US-11 同角色多会话（当前 / 历史 · 选择即当前）**：作为 Owner，当某角色下有多个会话时，我希望它们区分为**一个「当前会话」+ 若干「历史会话」**——当前默认时间最新；**我点哪个会话，哪个就成为当前**（选择即当前，无需额外操作），其余为历史、可回溯；以便像翻对话历史一样对照该角色不同时期的工作。
- **US-13 迭代时间轴（保留 v0.2 功能，置于项目会话第一级）**：作为 Owner，进「项目会话」选中某项目时，我希望看到该项目的**双层迭代时间轴**（迭代总览 + 阶段门禁，阶段可「查看对话」钻取），以掌握项目迭代进度——此为 **v0.2 已有功能**，v0.3 抽屉版重构对话查看器时**不得弄丢、须保留**，置于项目会话第一级（项目概览）；阶段「查看对话」则打开对话抽屉、定位到该阶段责任角色。

### B 组 · 匹配增强（解析侧治理，方案依 BCR-013）

- **US-4 角色识别改进 + 兜底 General**：作为 Owner，我希望会话的角色标签尽量识别准确，且**识别不出角色的会话统一归为 General**（而非悬空的 Unknown），以便会话列表没有"无角色"的孤儿项。
- **US-5 迭代归属重建 + 迭代标签**：作为 Owner，我希望每个会话能标出它**大致属于哪个迭代**（按会话活动时间 × 项目 `INDEX.md` git 历史重建），并在会话选择器里显示「所属迭代」标签，以便我在页内配会话时快速找到「v0.2 的 PM 会话」这类目标。
- **US-10 拖拽打标签纠错**：作为 Owner，当某会话角色被自动识别错时，我希望通过**拖拽**把它从当前角色拖到正确角色（拖到目标角色 Tab / 放置区）来改其角色标签（覆盖自动识别），直观地把会话归到正确角色，形成「自动识别 → General 兜底 → 拖拽纠错」完整闭环。
- **US-12 多来源会话同步（Codex）**：作为 Owner，我希望 workboard 除 Claude Code 会话外，也同步 **Codex** 的会话，让 Codex 会话一并进入匹配闭环（识别 / 打标签 / 归类 / 切换）并带**来源标签**区分，以便统管多来源 AI 会话。（Trae CN 存储加密无解，标记为已知边界、暂不支持。）

### C 组 · 菜单精简（方案 A，独立可砍）

- **US-6 顶级菜单 5→3**：作为 Owner，我希望顶级菜单从「工作台 / 部署 / 接入诊断 / 跨项目 / 项目会话」收敛为「看板 / 项目会话 / 需求池」，部署与接入诊断降级进看板项目卡片详情抽屉的 tab，以便菜单更聚焦、减少并列冗余。
- **US-8 需求池·沟通可查看内容**：作为 Owner，我希望在需求池的「沟通」tab 里点击某条沟通，能看到该沟通的完整内容，而不只是一句摘要，以便追溯跨项目沟通的来龙去脉。

### D 组 · 看板打磨

- **US-7 跨项目待办折叠**：作为 Owner，我希望看板的「跨项目待办」在条数较多时默认折叠（只显示前几条），鼠标 hover 能预览更多、点击可展开全部，以便看板不被长列表撑开、保持整洁美观。

### E 组 · 部署健壮性

- **US-9 生产友好错误页**：作为 Owner，我希望生产环境在后端不可用（500/502）或访问无效路径（404）时，显示与 workboard 风格一致的友好错误页，而非裸 nginx 报错页，以便（尤其参赛演示时）保持专业观感。

## 3. 验收标准

### US-1 对话视觉打磨
- 当打开任一会话对话时，应该以区分度更高的视觉呈现 user / assistant / 工具调用消息（如角色化配色、头像或标识、消息间留白与分隔），而非仅靠两种底色色块区分。
- 当消息正文含结构化内容（代码块 / 列表 / 标题等 Markdown）时，应该以可读的格式呈现，而非整段纯文本平铺（最低要求：代码块等宽块状呈现）。
- 当对话内容为空或全为工具调用时，应该显示明确的占位提示，而非空白页。

### US-2 页内横向切角色
- 当我在对话查看器内点击另一个角色的 Tab 时，应该在**当前页面**加载该角色所映射会话的对话，页面不返回上一级。
- 当我切换到的角色**已映射会话**时，应该正确加载该会话内容；切换应有明确的当前选中态标识。
- 当某项目角色数量变化或某角色被禁用时，Tab 区应正确反映当前可切换的角色集合，不出现指向已失效映射的死 Tab。

### US-3 页内配会话（1:N）
- 当为某角色配置会话时，选择器应**只列识别为该角色的会话**（强限制）；General 列识别不出的会话；一个角色可归属**多个**会话。
- 当某角色下**还没有归属会话**时，应显示空态 + 内联会话选择入口，而非报错或空白；选中即把会话归入该角色。
- 当会话数据源不可用（如本地开发数据库未连）时，选择入口应显示明确的加载失败提示，而非静默无响应。
- 会话通过识别（`detectRole`）或手动打标签（US-10）归入角色；同一会话归属唯一角色（打标签 = 从原角色移出、归入新角色）。

### US-11 同角色多会话（当前 / 历史 · 选择即当前）
- 每个角色的会话应区分为**一个当前会话 + 若干历史会话**；当前**默认 = 时间最新**，其余为历史。
- 当我**点击某个会话**（当前或历史）时，它应**成为当前会话**（选择即当前），原当前降为历史、对话区加载它——**无需单独的「设为当前」按钮**。
- 「当前」为**前端临时状态、不持久化**——刷新 / 重开 / 切项目后恢复默认（当前 = 时间最新）；角色归属手动标签（`manualRole`）则**持久保留**（两者策略不同：查看偏好临时、事实纠正持久）。
- 每个角色**有且仅有一个**当前会话；当前应突出区别于历史（标「当前」）、历史「可回溯」。
- **会话行显示会话自身信息**（标题 / 时间 / 迭代标签 / 来源标签），**不显示角色**（左侧已选角色、再标冗余）。
- 单会话时即当前、无历史；无归属会话时显示空态选择入口（同 US-3）。

### US-13 迭代时间轴（保留 v0.2，项目会话第一级）
- 当进「项目会话」选中某项目时，应显示该项目的**双层迭代时间轴**：迭代总览（各迭代 / 版本 + 进度）+ 阶段门禁（PRD / 设计 / 实现 / 部署检查各阶段状态）。
- 当点击某阶段的「查看对话」时，应**打开对话抽屉**并定位到该阶段责任角色（复用 US-2/11 抽屉），而非旧的全屏对话查看器。
- 时间轴数据源沿用 v0.2（项目 `INDEX.md` 版本列表 + `vX.Y.md` 阶段门禁解析）；无迭代记录时显示空态提示。
- 此为**保留 v0.2 已有功能**（`IterationTimeline`）：实现复用 v0.2 组件，仅调整位置（项目会话第一级）与阶段钻取目标（改为打开抽屉）。

### US-4 角色识别改进 + 兜底 General
- 当某会话经 `detectRole` 无法判定具体工作流角色时，应该归类为 `General`，会话列表中不再出现 `Unknown` 标签。
- 当会话首条用户消息为「你是 {角色}」等明确信号时，应该正确识别为对应角色（不因兜底逻辑回归而误判为 General）。
- 在真实开发项目会话样本上，改进后开发角色会话的「识别不出具体角色」比例应不高于改进前（以基线 ~7.5% 为参照，不劣化）。

### US-5 迭代归属重建 + 迭代标签
- 当某会话可按「会话活动时间 × 项目 `INDEX.md` git 历史」定位到某迭代时，应该在会话选择器/会话展示中显示对应「所属迭代」标签（如 `v0.2`）。
- 当某会话**无法定位到具体迭代**（如活动时段该项目「当前迭代=无」、会话早于 INDEX、非工作流项目）时，应该显示为「未归属 / 未识别」之类的中性标注，而非错误地硬套一个迭代。
- 迭代标签应标明其为**推断结果**（如来源提示「按活动时间推断」），不呈现为绝对确定值。

### US-10 拖拽打标签纠错
- 当拖动某会话到另一个角色（目标角色 Tab / 放置区）并释放时，应把该会话角色标签改为目标角色（**覆盖**自动识别），会话随即归入目标角色分类（如从 General 拖到产品，则出现在产品 Tab 的会话选择器里）。
- 拖拽过程应有明确视觉反馈：拖起态、可放置目标高亮、释放确认。
- 手动标签应**持久化**（workboard 本地存储），刷新后保留；**不回写**被监控项目 / 会话源文件。
- 触屏或不便拖拽时，应有备选入口（如长按 / 右键菜单选角色）兜底改角色。

### US-12 多来源会话同步（Codex）
- 当本机有 Codex 会话时，应被同步入库、出现在会话池，与 Claude 会话一起参与角色归类 / 打标签 / 切换（US-3/10/11）。
- 每个会话应显示**来源标签**（Claude / Codex），可区分来源。
- Trae CN 因存储加密无法读取正文，应标记为「暂不支持」的已知边界，不报错、不误导。

### US-6 顶级菜单 5→3
- 当进入应用时，顶级菜单应显示为「看板 / 项目会话 / 需求池」三项。
- 原「部署」「接入诊断」的信息应可在看板项目卡片详情抽屉的对应 tab 内访问到，不丢失能力。
- 原「跨项目」的需求池 / BCR / 谁等谁 / 沟通等内容应在「需求池」入口下可访问（作为子 tab 或分区），不丢失能力。

### US-7 跨项目待办折叠
- 当跨项目待办不超过阈值（默认 5 条）时，应全部直接显示，不出现折叠控件。
- 当超过阈值时，应默认只显示前 5 条 + 底部「展开全部（还有 X 条）」入口。
- 当 hover 折叠入口时，应给出更多待办的预览/提示，且不破坏页面布局与其余区块位置。
- 当点击展开时，应内联展开全部（超高时可滚动），入口变「收起」；再点收起恢复折叠态。
- 当**单条待办文本过长**（一行显示不全）时，应默认截断（最多两行 + 省略号）；hover 显示完整内容（悬浮浮层，不挤动表格布局），或点击展开查看全部。此项与「列表条数折叠」为两个独立维度。

### US-8 需求池·沟通查看完整内容
- 当在需求池「沟通」tab 点击某条沟通时，应打开抽屉展示该沟通对应 `communications/{REQ-id}-*.md` 文档的**完整内容**（Markdown 渲染），而非仅一句摘要。
- 当对应沟通文档缺失或读取失败时，应显示明确提示（如「沟通文档未找到」），而非空白或裸错误。
- 文档只读展示、不可编辑（符合 workboard 只读定位）。

### US-9 生产友好错误页
- 当后端服务不可用（如 502 / 500）时，应返回与 workboard 视觉一致的友好错误页（如「服务暂时不可用」），而非 nginx 默认裸报错页。
- 当访问不存在的资源（404）时，应显示友好的 404 页，或由 SPA 兜底进 app 的「页面不存在」提示，不出现裸报错。
- 错误页须能在生产 nginx 配置中正确挂载（`error_page` 指向静态页），且视觉与 workboard 一致。

## 6. 风险和开放问题

- **静态快照假设的固有局限**（BCR-013 点破，供实现与验收知情）：「一会话 = 一角色一迭代」并不总成立——一个会话可能先关 v0.2 再启 v0.3、参谋长/General 会话不属任何迭代、迭代可能在会话中途推进。因此**迭代标签是「尽力而为的归属」而非绝对真相**，UI 须留余地（标来源、允许中性「未归属」态）。此为已知局限，不追求全覆盖。
- **迭代维命中率天花板**：基线摸底开发角色会话命中 62.8%，未命中多为结构性不可归属（当时无迭代/老会话）。开放问题：未命中会话的展示口径（统一「未归属」还是细分原因），设计阶段定。
- **生产环境 `.git` 可读性**：生产部署下各项目仓是否以可读 `.git` 形态存在，需 DevOps / Architect 在部署就绪检查前确认；若不可读，迭代维需降级方案（开放项）。
- **`detectRole` 扫 assistant 消息的收益**：作为角色维可选增强（进一步降 Unknown），其成本/收益由 Developer 在设计/实现阶段评估，非必做；兜底 `General` 为必做项。
- **菜单 5→3 量级**：C 组是本迭代唯一的信息架构重构，量级最大、与 A/B 主题不同。已约定**独立成组、验收独立、实现最后做**，必要时可单独砍出而不影响 A/B 主线。
- **hover 浮层被 `overflow-hidden` 裁剪**（US-7 实现注意）：`CrossTodoList` 外层 `overflow-hidden`（为圆角裁切）会裁掉向下弹出的 hover 浮层——待办全文浮层与折叠 peek 预览都是 `absolute top-full`，末几行/底部按钮的浮层会溢出容器被裁。design 稿用少量示例数据看不出；实现时 Developer 须处理（浮层 portal 到 body、或改圆角实现方式），确保浮层完整可见。
- **沟通文档路径映射**（US-8）：`commItems` 的 `reqId` → `communications/{REQ-id}-短名.md`，文件名含「短名」需匹配（按 `REQ-id` 前缀扫目录定位），设计阶段定映射规则；找不到时按验收走「未找到」提示。
- **`coordination.test.js` 假失败**（跨任务待办已登记）：与本迭代无直接关系，但若实现阶段触碰会话/解析测试，可顺带评估是否 fixture 化（归 Developer 判断，不强绑本迭代范围）。

## 7. 实现指引（design 蓝本 → frontend 复现路径）

> 给实现阶段 Developer：如何用 design 原型图 + 本 PRD 落到 `frontend/` 真代码。**核心：翻译不是复制；后端从零。**

### design 原型图的定位
claude.ai/design 项目 `niuma-cheng-workboard` 的组件是**高保真视觉 + 交互蓝本**，**不是可直接部署的生产代码**——它跑在 design 沙盒（`window.React`、design 自建 `lib/*` 与 primitives、`sampleData` 演示数据），与 `frontend/`（Vite + 标准 import + `lucide-react` + shadcn/ui + 真实 `/api/snapshot`）**四层不兼容，照搬跑不起来**。**原型图留存、别弃**——它是实现阶段的视觉蓝本。

### 前端：翻译，不是复制（中等工作量）
Developer 用 **DesignSync 读出 design 组件完整代码**当蓝本，翻译成 frontend 生产组件：
- `const React = window.React` → 标准 `import`；
- design `lib/icons.jsx` / `cn.js` / 自建 primitives → `lucide-react` / shadcn/ui `@/components/ui/*`；
- `sampleData` → 真实 snapshot 数据；
- **保留 Tailwind `className` + 交互逻辑**——frontend 同用 Tailwind + 同套 token，视觉与交互即高度还原。

### 后端：从零做（工作量大头，design 无后端）
design 是纯前端 mock，**所有数据管道要从零实现**（详见 §5 前置依赖）：
- US-11 `manualRole` 本地持久化（当前会话 `currentByRole` 前端临时、不持久）+ v0.2 的 1:1→1:N 迁移；
- US-12 Codex 解析器（`codex-parser.js` 现状核实）+ schema `source` 列；
- US-8 `communications/*.md` 读取 + Markdown 渲染；
- US-5 INDEX-git-history 重建；US-4 `detectRole` 改进 + 兜底 General。

### 工作量判断
**复现 ≈ 前端翻译（中等）+ 后端从零（大头）。** 原型图完整的是「长什么样」，不是「数据怎么来」——真正成本在后端，Developer 评估工程成本时以此为准。
