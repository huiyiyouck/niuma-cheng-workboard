## P0 关键：7.3「承接方匹配 id」规则与实际数据对不上

文档 7.3 / 6.1 / Review 第 12 项确定第一版「按 `承接方` 字段匹配配置项目 `id`」。但 `coordination/REQUESTS.md` 实际数据为：

- 承接方：`ai（承接人待 Bootstrap 后补登 PM/Architect）`
- 提出方：`xiaobao · Developer`

即 `承接方` 是**带括号注释的自由文本**，不是干净 `id`，直接等值匹配 `承接方 === "ai"` 会失败。该条虽已被 Owner 确认（第 9/12 条），但实现细节有坑，需在文档明确口径，二选一：

1. **看板侧**：改为「在 `承接方` 文本中按词边界包含项目 `id`」。注意 `ai` 这类短 id 易在其它词中误命中（substring 风险），需做 token / 词边界切分。
2. **数据侧（更稳，推荐）**：在 coordination 约定 `承接方` 单元格以纯 `id` 开头，或单列结构化 `承接方 id`，把自由说明移到备注列。与第 8 节 frontmatter 思路一致，且应「先在 agent-workflow / coordination 定规则再同步」。

附注：`提出方` 为 `xiaobao · Developer`（项目·角色）格式，7.3 已把「匹配提出方」列为后续扩展，方向正确，但同样需按此格式做前缀切分。

## 待确认项评估（请 codex 确认）

针对草案 Review 表中仍为「待确认」的第 11、13、14、15、16 项，Reviewer 评估结论与建议如下。**请 codex 逐条确认；确认后再据此更新草案对应章节与 Review 表状态。**

### 11. coordination 总览只展示摘要，不要求 INDEX.md
**评估：建议确认。** coordination 是跨项目真源仓库，不是业务项目，强行要求 `docs/progress/INDEX.md` 属错误耦合；其内容本就在 `PROJECTS.md` / `REQUESTS.md` / `STATUS.md` / `contracts/` / `communications/` 中。

补充口径（确认后写入 3.3 / 4.1 第 1 条）——四个摘要计数必须写死：
- 活跃需求数 = `REQUESTS.md` 中未关闭状态的行数
- 跨项目阻塞数 = `STATUS.md`「跨项目阻塞与谁等谁」中的有效阻塞条目（无阻塞记 0）
- 契约数 = `contracts/*.md` 文件数
- 沟通文档数 = `communications/*.md` 文件数

### 13. 入口链接由看板自动生成（INDEX.md 不单独写链接）
**评估：建议确认。** 各项目手写链接冗余且易过期，自动生成更省心，符合「配置驱动、看板负责格式化」的定位。

补充口径（确认后写入 7.1）：
- 「当前迭代文档」链接需先从 INDEX.md 解析当前迭代号再拼路径，存在解析依赖；**需加回退**：拼不出时降级到 `docs/progress/` 目录链接，不得报错。
- 看板为本地只读服务，需明确链接形式（本地文件路径 vs 服务内路由），避免点击打不开。

### 14. 新增项目必须先写 `projects.config.json` + 通过校验，不自动扫描
**评估：强烈建议确认，无坑。** 这是整个设计的基石。自动扫目录会误纳入临时 / 非项目目录，且与「配置驱动、不硬编码」直接冲突。显式接入 + 校验是正确选择。

### 15. 第一版必须提供「接入配置监控页」
**评估：建议确认，但放宽实现形态。** 配置驱动是核心、项目数会增长、路径解析与 `${PROJECT_HOME}` 展开易出错，需要能回答「这个项目为什么没出现」的诊断视图，价值高，值得进第一版。

建议（确认后写入 6.6）：把「页面」放宽为「**接入诊断视图**」——第一版不必做独立精致页面，可作为总览页内的诊断区块 / 表格，复用同一份解析结果即可。确认「必须有接入诊断展示」，实现形态不锁死。

### 16. 接入配置页面只读
**评估：强烈建议确认，无坑。** 页面可写配置会牵出文件写入、路径安全、误操作回滚，第一版只读是正确的最小风险边界。草案 6.6 已写明理由。
