# agent-workflow 多角色协作基线

| 字段 | 值 |
|------|-----|
| 版本 | v1.0 |
| 状态 | 通用模板 |
| 适用范围 | 使用 Codex、Claude Code 或其他 AI 编程助手的单人/小型项目 |

## 1. 目标

这套基线把 AI 助手会话组织成一个轻量开发团队：同一个人可以在不同时间以 PM（产品经理）、Architect（架构师）、Developer（开发工程师）、DevOps（运维/部署工程师）等角色启动 Agent，让项目拥有启动、需求、设计、实现（含自测）、部署和复盘的持续记忆。界面要点并入 PRD（PM 负责）；质量由 Developer 自测 + Owner 验收保障。

人类用户是项目 Owner（负责人）和实际项目经理，负责最终协调、优先级判断和流程取舍。工作流不设置常驻 Project Manager（项目经理）Agent，避免让一人公司产生没有实际价值的管理层。

它不是企业级流程管理系统。它的重点是：少量文件、明确职责、可追溯状态、可复用模板。

## 3. 角色名称

文档中首次出现角色时应优先使用“英文代号（中文名称）”格式，方便阅读，同时保留稳定角色 id。

### 入口角色

入口角色只负责默认响应和角色切换，不属于工作流角色集合，不参与标准迭代门禁，不写角色日志。

| 英文代号 | 中文名称 | 角色手册 |
|----------|----------|----------|
| General | 通用助手 | 默认不读取；维护入口规则时可参考 `docs/baseline/role-general.md` |

### 工作流角色

工作流角色在用户明确或确认切换后才加载 `docs/baseline/runtime.md`，并按角色手册、当前状态和当前任务按需读取上下文。

| 英文代号 | 中文名称 |
|----------|----------|
| PM | 产品经理（含界面要点） |
| Architect | 架构师 |
| Developer | 开发工程师（含自测） |
| DevOps | 运维/部署工程师 |

> 角色集精简（BCR-004/006，2026-06-25）：原 UI（界面设计师）并入 PM、Tester（测试工程师）并入 Developer 自测 + Owner 验收，两份角色手册已墓碑化（`role-ui.md` / `role-tester.md` 标 `<!-- RETIRED -->`，见 role-pm / role-developer）。

## 4. 工作模式

本工作流支持标准迭代和非迭代工作两类模式。标准迭代用于明确要落地的产品功能；Bugfix、线上故障、产品方案、UI 草案、技术预研、运维任务等不强制进入迭代。

具体模式选择、记录路径和升级规则见 `docs/baseline/work-modes.md`。

## 5. 团队知识库

团队知识库用于沉淀可复用知识、未落地机会、架构取舍、Bug 根因、部署经验和复盘结论。它不是流水账，不能替代角色日志。

知识库路径为 `docs/knowledge/`。写入规则、分类和读取规则见 `docs/baseline/knowledge-base.md`。

## 6. 上下文治理

随着迭代次数增加，角色日志和 Review 记录会快速膨胀。Agent 启动时只读索引、摘要和相关文件，不读完整历史。

日志摘要、归档触发条件和读取层级见 `docs/baseline/context-policy.md`。

## 7. 标准迭代流水线

Bootstrap 初始化不是标准迭代本身。它只安装团队工作台。只有用户明确选择标准迭代后，才进入下面的流水线。

标准迭代只能从 PM（产品经理）创建 PRD 开始。其他角色可以提出迭代建议、记录机会、做技术预研或非迭代任务，但不能绕过 PRD 直接启动架构、开发或部署阶段。

```text
PRD 阶段（含界面要点） -> 设计阶段 -> 实现阶段（含 Developer 自测） -> 部署就绪检查 -> 迭代关闭检查（含 Owner 验收） -> 收尾归档
```

| 阶段 | 产出方 | Review 方 | 定稿条件 |
|------|--------|-----------|----------|
| PRD（含界面要点） | PM（产品经理） | 产出方按影响领域动态指定 | 指定 Review 方全部通过 |
| 设计 | Architect（架构师） | 产出方按影响领域动态指定 | 指定 Review 方全部通过 |
| 实现（含自测） | Developer（开发工程师） | 产出方按影响领域动态指定；验收/边界/回归由 Architect 或 DevOps 复核 | 指定 Review 方全部通过，且自测证据齐备 |
| 部署 | DevOps（运维/部署工程师） | 默认独立检查；高风险时动态指定确认方 | 部署检查通过，或明确记录阻塞/跳过原因 |
| 迭代关闭 | 当前 Agent 执行关闭检查；以 Developer 自测结论 + **Owner 验收**为前置，无法判断时报 Owner | 用户确认 | INDEX、迭代记录、当前角色日志、Change Note、知识沉淀、Owner 验收状态一致；其他角色只登记待补充 |

当前阶段未定稿前，不进入下一阶段。Spike 或纯技术预研可以跳过部分阶段，但必须在迭代记录中说明原因和实际 Review 方。

**阶段回环规则：** 如果后续阶段发现前置阶段产出有问题需要修改，由被修改阶段的产出方判断变更是否影响下游已定稿内容。受影响的下游阶段需重新确认（不自动全部作废，也不自动全部保留）；不受影响的保持定稿。判断结果写入迭代记录。

PM（产品经理）负责产品内容是否正确，但不代表人类做最终项目协调。阶段推进、优先级取舍、是否接受风险延期，由人类 Owner 最终确认。

Bootstrap 初始化、收尾归档、迭代关闭检查不是角色，而是非角色机制。触发时机、执行者和结束条件见 `docs/baseline/mechanisms.md`。流程审计由当前会话 Agent 执行；如发现工作流规则本身需要修改，按 §14 写 `BCR-###` 入 coordination 基线修正提案池，不在本项目直接改框架。关键结果必须由用户确认。

## 8. 状态词命名空间

不同层级使用不同状态词，不允许混用。角色更新状态时，必须先确认自己正在修改的是阶段门禁、文档生命周期、任务记录、归档记录、部署检查还是 Change Note。

### 8.1 阶段门禁状态

标准迭代各阶段门禁使用以下状态词：

| 状态 | 含义 |
|------|------|
| 待Review | 初版刚提交，尚无人 Review |
| Review中 | Review 方正在反馈，或等待部分 Review 方反馈 |
| 修改中 | 产出方正在处理 Review 问题 |
| 已定稿 | 当前轮次指定 Review 方均通过 |
| 阻塞 | 需要人类或外部条件解除 |
| 已跳过 | 该阶段不适用于本迭代，必须写原因 |

### 8.2 其他状态词

| 层级 | 允许状态 | 真源 |
|------|----------|------|
| 迭代最终状态 | 进行中 / 已完成 / 已关闭 / 阻塞 | `docs/progress/iterations/vX.Y.md` |
| 非迭代任务状态 | 进行中 / 已完成 / 已中止 / 升级为迭代 / 已归档 | `docs/progress/ad-hoc/` |
| 收尾归档状态 | 未收尾 / 收尾中 / 已归档 | `docs/progress/iterations/vX.Y.md` 或对应 ad-hoc |
| 部署检查状态 | 待检查 / 检查中 / 通过 / 阻塞 / 已跳过 | 迭代记录的部署就绪检查 |
| Owner 验收状态 | 未验收 / 通过 / 打回 / 有条件通过 | 迭代记录关闭归档区「Owner 验收」小节 |
| Change Note 状态 | 待确认 / 已确认 / 已执行 / 已归档 / 已废弃 / 升级为重大变更 | Change Note 文件和迭代记录 |
| 跨任务待办状态 | 待处理 / 处理中 / 待确认 / 已完成 / 已废弃 | `docs/progress/INDEX.md` 跨任务待办 |

`已定稿` 只用于阶段门禁；`已完成` 只用于任务或迭代结果；`已归档` 只用于收尾、Change Note 或非迭代记录归档。不要用 `已完成` 代替阶段 `已定稿`，也不要用 `已归档` 代替迭代关闭。

### 8.3 状态真源规则

状态真源规则：

- `docs/progress/iterations/vX.Y.md` 是迭代阶段门禁和 Review 轮次的真源。
- `docs/progress/INDEX.md` 是项目级当前状态和下一步入口的真源。
- `docs/baseline/project-context.md` 只写项目事实，不写当前阶段或 Review 状态。
- PRD、设计文档、自测报告头部可以保留状态摘要，但定稿前必须检查它们和迭代记录一致。
- 任何角色改变阶段状态时，必须先更新迭代记录，再同步 `INDEX.md`；不要只改自己的产出物。

## 9. 动态 Review 计划

标准迭代不采用全员固定 Review。每个产出方在提交产出物时，必须根据本次变更影响领域动态指定 Review 方，并写明指定理由和未指定理由。

### 9.1 选择维度

| 影响领域 | 应考虑的 Review 方 |
|----------|--------------------|
| 产品范围、用户故事、验收标准、用户流程/页面/交互/视觉 | PM（产品经理） |
| 架构、数据流、接口、技术选型 | Architect（架构师） |
| 实现复杂度、工程成本、兼容性 | Developer（开发工程师） |
| 验收标准、边界条件、回归风险 | Architect（架构师）或 DevOps（运维/部署工程师）——**验收标准由 PM 产出时，PM 不复核自己产出的验收标准** |
| 部署方式、环境变量、云服务、发布风险 | DevOps（运维/部署工程师） |

例如：全新项目通常应指定 Architect（架构师）；界面要点在 PRD 中由 PM 覆盖；部署方式从本地改为云端，应指定 DevOps（运维/部署工程师）。

> **下游实现方进设计文档 Review（条件触发）：** 当设计文档涉及前后端子 Agent **并行实现**（见 `role-developer-detail.md` 子 Agent 调度）时，应把下游实现方 Developer（开发工程师）列入该设计文档的 Review 方——让消费者在自己实现会话开始**之前**，基于接口契约提出待澄清问题，答案在定稿前回填正文（见 §10.1）。不涉及并行实现的设计文档不强制纳入，避免无谓增加 Review 方。

### 9.2 最低质量底线

- 标准迭代的核心产出默认至少指定 2 个 Review 方。
- 如果少于 2 个 Review 方，产出方必须写明原因，并由用户确认。
- Review 方不是越多越好；只指定真正受影响或能提供关键判断的角色。
- 定稿条件是：本轮 Review 计划中指定的 Review 方全部完成并通过，且无未处理的高严重度阻塞问题。

### 9.3 Review 计划模板

见 `docs/templates/review-plan.md`。

## 10. Review 轮次

每个阶段按 `R1`、`R2`、`R3` 递增。

- `R1`: 初版提交。
- 有任何 `需修改`：产出方进入 `修改中`，修完后提交 `R2`。
- 本轮指定 Review 方全部 `通过`：产出方将阶段改为 `已定稿`。
- 连续两轮 Review（R3）仍未通过：升级为 `阻塞`，由用户直接决策（接受现状/修改需求/关闭迭代），不进入 R4。

Review 记录必须包含：轮次、结论、问题、严重程度、建议。不要只写”感觉不行”。

### 10.1 待澄清问题与缺陷分流

Review 方基于产出物提出的「问题」分两类，处置不同：

| 类型 | 含义 | 处置 |
|------|------|------|
| 缺陷 | 产出有错 / 有漏，按严重度分级（见下） | 产出方进入 `修改中` 改正文，走轮次流程 |
| 待澄清问题 | 不是错，是文档未写清、下游实现会卡住的歧义 | 产出方逐条作答，**答案回填文档对应正文**（数据模型 / 接口契约 / 核心流程），Review 记录只留指针；不计严重度 |

**答案回填正文是底线：** 隔离的下游实现会话（及其并行子 Agent）只读文档正文，读不到 Review 线程。澄清答案若只留在 Review 记录而不回填正文，等于没交接——下游照样拿不到。澄清问题作答属补全正文，不属「Review 阶段不得改正文」禁止之列（后者禁的是绕过 Review 静默改结论），但须在 Review 记录留回填指针、可追溯。

> **一人场景提示：** 同一人类以不同角色 Review 自己的产出时，应在切换角色后有意识地切换思维模式——PM 审”需求对不对”，Architect 审”架构能不能撑住、边界条件有没有漏”，DevOps 审”部署与回归风险”。机械走流程而不切换视角，Review 会失去价值。

### 缺陷严重度

| 严重度 | 定义 | 动作 |
|--------|------|------|
| 阻塞 | 核心功能不可用、数据丢失、安全漏洞 | 必须先修，标记 `阻塞` |
| 高 | 主要流程中断但可绕过 | 本迭代必须修 |
| 中 | 边界条件问题或次要功能异常 | 可延期，需记录原因 |
| 低 | 体验瑕疵、文案错误 | 不阻塞定稿 |

## 11. 定稿后问题处理

文档定稿后，未参与 Review 的角色或后续执行角色可能发现问题。原则上不直接修改已定稿正文，而按影响程度分三档处理。

| 档位 | 适用情况 | 处理方式 |
|------|----------|----------|
| 实现取舍 | 方案可实现，但不是最优；不改变产品范围、接口契约、UI 验收或部署方式 | 不改定稿文档，在实现记录或角色日志中写明取舍，必要时沉淀到知识库 |
| 轻量变更 | 需要补字段、补边界说明、调整小范围验收或接口细节，但不改变阶段大方向 | 创建 Change Note，由受影响角色确认，不重启完整 Review |
| 重大变更 | 产品范围、主流程、架构方案、部署方式、关键验收发生变化 | 创建变更提案，回到对应阶段重新 Review |

### 11.1 Change Note 流程

Change Note 详细执行规则和归档流程见 `docs/baseline/mechanisms.md`。

### 11.2 定稿后问题的处理底线

- 不能静默修改已定稿正文。
- 不能因为“只是 Agent 自己实现起来麻烦”就重开完整 Review。
- 不能把重大变更伪装成轻量变更。
- 允许实现角色在不改变契约的前提下做实现取舍，但必须写入实现记录或角色日志。
- 允许轻量变更在受影响角色确认后直接执行，但必须可追溯、可回看、可归档。

## 12. 写权限

| 文件 | 谁能改正文 | 谁能追加 |
|------|------------|----------|
| PRD | PM（产品经理） | Review 计划中指定的角色追加 Review |
| 设计文档 | Architect（架构师） | Review 计划中指定的角色追加 Review |
| 代码 | Developer（开发工程师） | Review 方只追加 Review，不直接改正文；需改正文由产出方进入 `修改中` 处理，或退出 Review 后按 Change Note / 回阶段规则执行 |
| 自测报告 | Developer（开发工程师） | Review 计划中指定的角色追加 Review |

| 非迭代工作记录 | 执行角色 | 其他角色只在被邀请时追加意见 |
| 团队知识库 | 各角色按领域维护 | 其他角色可补充来源和交叉引用 |
| 迭代记录 | 各阶段相关角色按职责更新 | Review 方追加自己的 Review 记录 |
| 角色日志 | 对应角色 | 无 |
| baseline/ | 经过人类确认后指定角色修改 | 任何角色可提出提案 |

## 13. 角色日志和纠错

每个角色维护：

```text
docs/progress/roles/{role}.md
docs/progress/roles/{role}-corrections.md
```

角色日志记录本次做了什么。纠错记录记录本角色犯过什么流程错误、如何避免复发。纠错记录保持 30 条以内。

角色日志超过上下文阈值时，必须按 `docs/baseline/context-policy.md` 做摘要归档。长期有价值的信息应提炼到 `docs/knowledge/`，不要只埋在日志里。

## 14. 基线修正

任何角色发现流程规则有问题，不能直接改 `baseline/`，先在角色日志中记诉求，再按 `cross-project-collaboration.md §基线修正提案流转（BCR）` 写 `BCR-###` 入 coordination `REQUESTS.md` 基线修正提案池：

```text
[基线修正提案] 问题：...；建议：...  → 转写为 coordination BCR-###（问题 + 拟改文件 + 理由）
```

`baseline/` 是工作流框架，由真源仓库统一维护。本项目内任何角色都不直接改 `baseline/`，只提 `BCR-###`，由 Owner + 参谋长评估、落地，再经 `sync-downstream.sh` 回流。

## 15. 角色创建机制

角色创建是可行的，但必须受控。坏的角色会增加流程噪音，让一人公司变成“一个人在管理一堆虚假岗位”。

新增角色必须满足至少一条：

- 有独立产出物。
- 有独立 Review 视角。
- 有明确不可替代的专业边界。
- 能减少现有角色的认知负担。

不应该新增角色的情况：

- 只是把一个任务换个名字。
- 职责和现有角色高度重叠。
- 没有产出路径、状态入口或退出条件。
- 增加的协调成本大于收益。
