# 标准迭代速查（standard-iteration-quick）

> 标准迭代必读的协议骨架。**仅标准迭代用**；跨模式安全红线在 `runtime.md`。每节标注的 §X 指向 `multi-agent-workflow.md` 完整规范，按需深读。

## 流水线（§7）

`PRD → UI → 设计 → 实现 → 测试 → 部署检查 → 迭代关闭 → 收尾`

- 只能 PM 创建 PRD 启动；其他角色只能提迭代建议，非 PM 要启动须先转 PM。
- 当前阶段未定稿不进下一阶段；Spike 可跳阶段但须在迭代记录写原因与实际 Review 方。
- 阶段回环：前置产出改动，由被改阶段产出方判断是否影响下游；受影响下游重新确认，其余保持定稿，结果写迭代记录。

## 状态机与真源（§8）

- 阶段门禁：`待Review / Review中 / 修改中 / 已定稿 / 阻塞 / 已跳过`。
- 命名空间不混用：`已定稿`仅阶段门禁、`已完成`仅任务/迭代、`已归档`仅收尾/Change Note/非迭代。
- 真源：`vX.Y.md` = 阶段门禁与 Review 轮次；`INDEX.md` = 项目级状态与下一步。改状态先更新迭代记录再同步 INDEX，不只改自己产出物。

## 动态 Review（§9-10）

- 按影响领域指定 Review 方：产品→PM、流程/视觉→UI、架构/接口→Architect、工程成本→Developer、验收/回归→Tester、部署→DevOps。
- 核心产出默认 ≥2 方；少于 2 须写原因 + 用户确认。只指定真正受影响的角色。
- 轮次 `R1→R2→R3`；R3 仍未通过升级`阻塞`交用户决策，不进 R4。Review 记录须含轮次/结论/问题/严重度/建议。
- 缺陷严重度：阻塞(先修) / 高(本迭代修) / 中(可延期记原因) / 低(不阻塞定稿)。

## 定稿后变更三档（§11）

- 实现取舍（不改契约）→ 不改定稿，写实现记录/角色日志。
- 轻量变更（补字段/边界/小范围验收）→ Change Note + 受影响角色确认，流程见 `mechanisms.md`。
- 重大变更（范围/主流程/架构/部署/关键验收变化）→ 回对应阶段重新 Review。
- 不静默改已定稿正文；不把重大变更伪装成轻量。

## 写权限与日志（§12-13）

- 正文：PRD→PM、设计→Architect、UI 方案→UI、代码→Developer、测试报告→Tester、迭代记录→各阶段角色。Review 方只追加 Review、不改正文；需改正文由产出方进入 `修改中` 处理，或退出 Review 后按 Change Note / 回阶段规则执行。
- 会话结束至少更新对应角色日志；状态影响项目入口时同步 `INDEX.md`。
